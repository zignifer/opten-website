## Schema Score: 69/100

Разметка opten.space технически зрелая: JSON-LD присутствует в исходном HTML, валиден и согласован с canonical/hreflang. Потери баллов вызваны не отсутствием schema, а семантическими связями между сущностями: author node, developer/vendor, исходные YouTube-видео и локальные `@id`.

Оценка строгая. Отсутствие `SearchAction` не является практической Google-проблемой, потому что Google прекратил поддержку sitelinks search box; этот пункт не следует внедрять только ради балла.

## High-priority findings

### SC-01 — Article author не развёрнут в локальном graph

**Статус:** подтверждено кодом и live JSON-LD.

На большинстве Blog, Model и Learn pages `author` ссылается на founder `@id`, но Person node с этим `@id` не всегда присутствует в graph конкретной страницы. Стабильный глобальный идентификатор полезен, однако isolated page graph должен позволять разрешить автора без предположения, что crawler уже видел главную/About.

**Действие:**

- использовать один стабильный author URL/`@id`;
- добавить полноценный `Person` node на страницы с human-authored content или обеспечить доступную author URL;
- видимую byline связать с `/about`;
- одинаково транслитерировать имя на RU/EN без создания двух людей.

**Приёмка:** на каждой Article/TechArticle/LearningResource странице author URL открывается и в graph существует разрешимая Person entity с `name`, `url`, `sameAs` и связью с Organization.

### SC-02 — model `SoftwareApplication` смешивает типы отношений

**Статус:** подтверждено в `scripts/seo-routes.ts` и prerender.

Model entity использует `brand` и `isRelatedTo` там, где их Schema.org domain не соответствует выбранному типу. `TechArticle`/`WebPage` при этом не связываются с локальным `#model`. На FLUX-страницах Replicate/fal.ai могут выглядеть как developer/vendor модели, хотя разработчиком является Black Forest Labs, а перечисленные сервисы являются hosting/access platforms.

**Действие:**

- определить, что именно описывает `#model`: software application, model/service или article subject;
- связать Article/WebPage с моделью через подходящее свойство вроде `about`/`mainEntity`;
- developer указывать только для реального разработчика;
- hosting/access platforms вынести в отдельный список предложений/провайдеров или видимый текст;
- удалить свойства вне допустимого domain вместо сохранения их как «похожих по смыслу».

**Приёмка:** Schema.org validator не показывает domain warnings; developer/vendor совпадает с официальным источником; Article имеет однозначную связь с model entity.

### SC-03 — сторонние YouTube-видео приписаны Opten

**Статус:** подтверждено на Learn Finds.

Learn Finds корректно показывают стороннего автора в интерфейсе, но `VideoObject.publisher` указывает Opten. Это смешивает издателя исходного видео и издателя редакторского конспекта. Opten является publisher страницы/анализа, но не стороннего YouTube-видео.

Также найдены связанные дефекты:

- у `/learn/ai-marketplace-product-cards` `uploadDate` в schema — 20.05.2026, тогда как дата source video — 08.07.2025;
- `inLanguage` меняется вместе с RU/EN страницей, хотя embedded source video остаётся тем же;
- Learn Find author/publisher должны описывать source creator отдельно от редактора Opten.

**Действие:**

- `VideoObject.author`/`publisher` брать из оригинального YouTube source;
- Organization Opten оставлять publisher/editor страницы и конспекта;
- `uploadDate` брать из source video metadata;
- `inLanguage` описывает язык видео, а не язык surrounding page;
- локализованный summary оформлять как отдельный WebPage/LearningResource.

**Приёмка:** три выборочных Finds совпадают с source YouTube по creator, uploadDate и language; Opten нигде не выглядит владельцем чужого видео.

## Medium-priority findings

### SC-04 — неполный image layer

У Article на model/about pages не всегда есть `image`. Это не делает schema невалидной, но ослабляет Article entity и multimodal presentation.

**Действие:** использовать реальное репрезентативное изображение страницы с абсолютным URL, корректными размерами и доступностью; не подставлять один generic logo вместо editorial image.

### SC-05 — Organization entity недостаточно отделена от founder

У Organization только три `sameAs`, причём личные YouTube/Telegram ссылки повторяются у Person. `Person.knowsAbout` присутствует, а Organization `knowsAbout` отсутствует.

**Действие:**

- личные profiles оставить у Person;
- Organization связывать только с реальными branded profiles;
- добавить `founder` на Person `@id`;
- использовать `knowsAbout` только для устойчивых продуктовых компетенций, которые подтверждены видимым контентом;
- закрепить единый `alternateName` и product descriptor.

### SC-06 — `speakable` указывает на отсутствующие селекторы

Homepage `SpeakableSpecification` ссылается на `.faq-question` и `.faq-answer`, которых нет в rendered HTML. Crawler не сможет разрешить указанные участки.

**Действие:** обновить `cssSelector` на реально существующие стабильные классы/атрибуты либо удалить блок, если speakable-поверхность не поддерживается как продуктовая цель.

### SC-07 — Learn breadcrumb hierarchy

Learn Find breadcrumb создаёт неоднозначный уровень и дублирование между `/learn`, Courses и Finds. Breadcrumb должен описывать фактическую публичную иерархию, а hidden course не должен становиться родителем public Find.

**Действие:** использовать `Главная → Learn → Находки → Материал` только если публичный индексируемый Finds hub действительно существует; иначе `Главная → Learn → Материал`.

### SC-08 — EN keywords содержит кириллицу

В EN Learn Find keywords остаётся строка `Оригинал на YouTube`. Это небольшой entity/language consistency дефект.

**Действие:** локализовать keyword source label или удалить его из keywords, если это UI-служебная фраза, а не тема страницы.

## What is already strong

1. Schema находится в исходном HTML, а не добавляется поздним client JavaScript.
2. Стабильные `@id` уже создают основу для Organization, Person, WebSite и SoftwareApplication graph.
3. FAQ content и `FAQPage.mainEntity` синхронизированы build-time gate.
4. Article/BlogPosting/HowTo/TechArticle покрывают основные редакционные шаблоны.
5. RU/EN canonical и hreflang не конфликтуют со schema URL.
6. URL внутри JSON-LD в проверенной выборке не ведут на 404.

6. Повторить URL integrity check для schema URLs.

## Ограничения

- Score отражает Schema.org/GEO completeness, а не вероятность конкретного rich result.
- Google не поддерживает все Schema.org properties одинаково; отсутствие Google feature не делает валидное свойство бесполезным для AI/entity extraction.
- Rich result не гарантируется даже при валидной разметке.
- SearchAction не рекомендуется возвращать только ради score: Google sitelinks search box retired.
- Продуктовый код в ходе аудита не изменялся.
