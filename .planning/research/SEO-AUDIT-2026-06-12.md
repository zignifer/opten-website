# SEO Audit: Google indexing and organic growth

Date: 2026-06-12
Site: https://opten.space
Scope: Google Search Console API, Yandex Webmaster API, live sitemap/robots/headers/HTML, local `dist/` output, route/schema source files.

## Executive summary

The screenshot is not showing a classic "site is blocked" SEO failure. The dominant Google issue is crawl/index priority:

- GSC reports 165 not indexed pages.
- 160 of them are `Discovered - currently not indexed`: Google knows the URLs but has not crawled them yet.
- 2 are `Crawled - currently not indexed`.
- 2 are redirects.
- 1 is an alternate canonical.

The site is crawlable: live pages return 200, canonical tags are present, robots allow public pages, sitemap has no GSC errors, and important samples have prerendered HTML. Yandex confirms this from another crawler perspective: the same site has 163 searchable pages and sitemap errors count 0.

The problem is that Google has not yet decided the full 196-URL live sitemap deserves crawl/index budget. This is amplified by a recent fast expansion: programmatic model pages, daily blog posts, public Learn pages, and Learn Finds were added faster than the domain's current authority can support in Google.

The Search Console video email is a real structured-data warning, but it is not the cause of 160 discovered/not-indexed pages. The warning came from `VideoObject.uploadDate` being emitted as `YYYY-MM-DD` without time or timezone. This audit includes a code fix in `scripts/seo-routes.ts`: video upload dates now render as ISO DateTime with `+05:00`.

## Findings

### P0 - Fix VideoObject uploadDate

GSC email:

- Invalid datetime value for `uploadDate`
- `uploadDate` is missing a time zone

Root cause:

- `scripts/seo-routes.ts` emits date-only values:
  - `learnFindVideoObjectBlock`: `uploadDate: find.publishedAt`
  - `videoObjectBlock`: `uploadDate: lesson.publishedAt`
- Source values are `YYYY-MM-DD`.

Google's Video structured data documentation defines `uploadDate` as date and time in ISO 8601 and recommends timezone information:
https://developers.google.com/search/docs/appearance/structured-data/video

Fix implemented in this audit:

- Added `videoUploadDate(date)` in `scripts/seo-routes.ts`.
- Date-only values now render as `YYYY-MM-DDT00:00:00+05:00`.
- Existing full DateTime values are left unchanged.
- Keep `datePublished` and `dateModified` as dates unless another validator complains.
- `npm run build` passed and generated Learn/Finds HTML now contains timezone-aware `uploadDate`.
- After deploy, use Rich Results Test and GSC "Validate fix".

Impact:

- Fixes the Search Console enhancement warning.
- May improve video rich result eligibility.
- Will not automatically solve the 160 discovered/not-indexed URLs.

### P0 - Expand Learn and Learn Finds content

| Learn lessons | 12 | 304 | 333 | 378 |
| Learn Finds | 10 | 405 | 462 | 534 |
| Learn hub | 2 | 464 | 504 | 504 |

This is the core content gap. Google is being asked to index many Learn pages where the visible textual value is mostly a video, a short description, timestamps, short notes, and resource lists. For competitive topics, that is not enough to beat YouTube, official product blogs, Habr, Sostav, Skillbox, large media, and original creators.

Recommended Learn page target:

- 900-1500 words per Find/Lesson before expecting Google growth.
- Add original editorial sections:
  - what the video teaches in practical terms
  - who should use it and who should skip it
  - step-by-step workflow
  - commands/prompts from the lesson
  - failure modes and risk notes
  - Opten-specific prompt improvements
  - source attribution and what Opten added beyond the original video
  - FAQ with visible DOM parity

For third-party video pages, avoid "summary-only" pages. Google's helpful content guidance explicitly warns against mainly summarizing others without substantial added value:
https://developers.google.com/search/docs/fundamentals/creating-helpful-content

### P1 - Google crawl budget is not keeping up with URL growth

The site moved from a small marketing surface to nearly 200 URLs quickly. Google is indexing a few strong URLs and leaving most new pages discovered/unknown.

This is normal for a young or low-authority domain with many newly generated/content-template URLs. A sitemap is a discovery hint, not an indexing guarantee:
https://developers.google.com/search/docs/crawling-indexing/sitemaps/build-sitemap

Recommended fix:

- Create a submitted sitemap index with sectioned sitemaps:
  - `sitemap-core.xml`
  - `sitemap-blog.xml`
  - `sitemap-models.xml`
  - `sitemap-learn.xml`
  - `sitemap-learn-finds.xml`
- Submit the section sitemaps separately in GSC.
- This is not a ranking boost, but it makes GSC diagnostics actionable by section instead of one mixed 196-URL bucket.
- Keep the existing all-URL sitemap or convert it into a sitemap index.

### P2 - Replace `/app/learn*` client redirects with server redirects

Current live behavior:

- `/app/learn` returns `200` SPA shell with `X-Robots-Tag: noindex,nofollow`.
- React then navigates to `/learn`.

This is not the main indexing blocker because `/app/*` is noindex and not in sitemap, but it creates unnecessary crawl noise and does not match the project contract that legacy Learn paths should redirect.

Recommended fix:

- Add Vercel redirects for:
  - `/app/learn` -> `/learn`
  - `/app/learn-v2` -> `/learn`
  - `/app/learn/:slug` -> `/learn/:slug`
  - `/space/learn` -> `/learn`
  - `/space/learn/:slug` -> `/learn/:slug`

## Why Google is not showing Opten in top results yet

Main reasons:

1. Most new URLs are not indexed, so they cannot rank.
2. Query-level GSC data shows almost no non-brand visibility yet.
3. Learn pages are thin compared with existing search competitors.
4. Model pages are good length, but they are programmatic and compete with official model docs, YouTube, Habr, Reddit, Sostav, Skillbox, and generator/vendor sites.
5. Domain/entity authority is still weak: Chrome Web Store has 298 users and 2 ratings; external independent mentions are limited.
6. Google appears slower than Yandex for this site; Yandex already sees the bulk of the pages.
