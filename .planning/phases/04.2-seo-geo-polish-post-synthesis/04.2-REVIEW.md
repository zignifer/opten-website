## Warnings

### WR-01: `BING_VERIFICATION_TOKEN_TODO` placeholder ships to every prerendered route

**File:** `index.html:20`
**Issue:** The Bing Webmaster meta tag was committed with a literal placeholder value `BING_VERIFICATION_TOKEN_TODO`. Since `scripts/prerender.mjs` uses `index.html` as its template, this placeholder is spliced into all 16 prerendered HTML files in `dist/`. Bing's verification flow does treat non-matching `content` values as "no-op" (it just fails verification — no crawl-level penalty), so this is operationally safe. But:

1. There is no build-time gate (no `verify-bing-meta.mjs` step) preventing a deploy where the placeholder is still active. The wave-5 SUMMARY notes "until token is filled" as a manual follow-up, but the placeholder will quietly survive production deploys indefinitely until someone notices in Bing Webmaster.
2. If the user does fill in the token via `index.html` edit, every prerendered file picks it up next build — that part is fine. The risk is purely "we forget" — but for a one-shot setup task that's a real risk.

**Fix:** Either (a) gate the build with a check that exits non-zero when `content="BING_VERIFICATION_TOKEN_TODO"` is found in any `dist/**/index.html` after prerender runs in CI (not local), or (b) move the meta tag injection to `prerender.mjs` `applyMeta` and read the token from an env var (`VITE_BING_VERIFICATION` or `BING_TOKEN`), emitting the tag only when set:
```js
// scripts/prerender.mjs (sketch)
const bingToken = process.env.BING_VERIFICATION;
if (bingToken) html = html.replace(/<\/head>/, `    <meta name="msvalidate.01" content="${escapeAttr(bingToken)}" />\n  </head>`);
```
Either approach removes the placeholder lifetime risk. Approach (b) also keeps the secret out of git, which matches how `VITE_PADDLE_CLIENT_TOKEN` is handled.

### WR-02: First IndexNow ping after deploy can never succeed (timing race)

**File:** `scripts/indexnow.mjs:48-81`
**Issue:** `npm run build` runs on the Vercel build worker BEFORE the build artifact is uploaded to the edge. The script POSTs the URL list to `api.indexnow.org`, which then attempts to fetch `https://opten.space/<key>.txt` to verify ownership. On the very first deploy with a brand-new key (or whenever Vercel's edge cache for the keyfile is cold), IndexNow's verification fetch will hit either (a) the previous deploy (no keyfile yet), or (b) a 404. Result: a 4xx response from IndexNow. The script does handle this gracefully (logs a warning, never fails the build — confirmed at line 71-73), so it's not a build-breaker.

However, the committed key (`36ced30e188f58b51d26c30e483124d5`) has already been visible in git history since commit `4036068`, and once the keyfile is live in a deploy, subsequent pings will work. So this is a one-time gotcha that only matters if the key is ever rotated.

Two related sub-issues:

1. **Multiple keyfiles in `public/`** → `findKey()` does `.find(...)` which returns the first match in `readdir` order (filesystem-dependent on Windows vs Linux build workers). If a key rotation leaves both old + new keyfiles in `public/`, the script silently picks one. The IndexNow POST will succeed for whichever happens to match, fail for the other — best-effort warning swallows this. A second `.txt` matching `[a-f0-9]{8,128}` would also be a footgun (e.g., somebody drops an unrelated hex-named file).
2. **No length-1 assertion** on the `.find()` result — `if (files.filter(re).length !== 1)` would catch both "no key" and "multiple keys".

**Fix:**
```js
async function findKey() {
  const files = await readdir(PUBLIC);
  const matches = files.filter((f) => /^[a-f0-9]{8,128}\.txt$/.test(f));
  if (matches.length === 0) {
    throw new Error("indexnow: no key file in public/ ...");
  }
  if (matches.length > 1) {
    throw new Error(
      `indexnow: multiple key files in public/ (${matches.join(", ")}). Remove all but one before rotating.`,
    );
  }
  return matches[0].replace(".txt", "");
}
```
The first-deploy timing race is mostly fine to leave as-is given the best-effort design — but a one-line comment at the top of `indexnow.mjs` noting "first deploy after a key rotation may see a 4xx warning; this is expected" would save future-you 5 minutes.

### WR-03: `NotFound.tsx` useEffect re-runs on every language change because `t` is a fresh function each render

**File:** `src/app/pages/NotFound.tsx:51`
**Issue:** The effect declares `[t]` as its dependency array. `useT()` returns `useContext(LangContext).t`, and `LangProvider` constructs `value={{ lang, setLang, t }}` inline at line 145 of `LangContext.tsx`, so the `t` reference changes on every parent re-render. Every time `LangProvider` re-renders (which happens on `setEnReady`, `setLangState`, location change, etc.), this effect's cleanup runs and the effect re-runs:

1. Cleanup removes the meta tag we created (because `createdMeta=true` in the previous closure).
2. Effect re-runs: looks for `<meta name="robots">` — finds none (we just removed it) — creates a new one. Sets `previousTitle = document.title` which is the title we just restored. Sets new title.

The cleanup/inject cycle is correct (no state leaks because each closure tracks its own `createdMeta` and `previousTitle`), but it is **unnecessary churn on every parent re-render** for what should be a mount-only effect. There's also a brief sub-paint window where the meta is removed before re-injection — search engines won't observe it, but anything reading `document.head` synchronously during that window would see no robots meta.

Concretely: if a user lands on `/typo`, the `LangProvider`'s `useEffect` (lines 99-121 of `LangContext.tsx`) fires `setLangState(...)` on mount, which re-renders the provider, which gives `useT()` a new `t` identity, which re-runs the `NotFound` effect. So the cycle fires at least twice on every 404 visit (mount + post-detect re-render).

**Fix:** Either drop the dependency entirely (the strings are read at injection time and we don't care about live updates after mount — closing the title accurately enough for a 404), or read the strings once at the top of the effect into local variables and depend on `lang` only:
```tsx
const { lang } = useLang();
useEffect(() => {
  const title = t("notFound.title");
  const previousTitle = document.title;
  document.title = `${title} — Opten`;
  // ... meta injection ...
  return () => { /* cleanup */ };
  // eslint-disable-next-line react-hooks/exhaustive-deps
}, [lang]);
```
Or, since the body re-renders with `t()` anyway, just drop the title update from the effect (let the body's React tree be the source of truth for "what language am I in") and keep `[]` for the meta injection only.

**Fix:** None required; add a leading comment if it makes future-maintainer life easier.

### IN-02: `NotFound.tsx` returns 200 OK — already documented but worth surfacing

**File:** `src/app/pages/NotFound.tsx:5-6`
**Issue:** The component injects `noindex,nofollow` via meta tag, but the HTTP status is 200 OK (Vercel SPA rewrite serves `dist/index.html`). Soft-404 from a search-engine perspective: Google specifically warns that meta-noindex on a 200 response is weaker than an HTTP 404. The comment correctly notes this is deferred to Phase 6 architectural refactor.
**Fix:** None for this phase. When Phase 6 introduces a Vercel edge function or middleware for HTTP 404, also update this comment.
