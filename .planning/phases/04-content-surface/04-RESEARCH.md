## Summary

Phase 4 lays an indexable content + structured-data layer on top of the Phase 3 bilingual prerender pipeline. The work breaks into three loosely-coupled streams that share a single build-time injection point (`scripts/prerender.mjs`):

1. **Schema bundle** — eight schema.org types (`Organization`, `SoftwareApplication`, `WebSite`, `Product`, `FAQPage`, `HowTo`, `BreadcrumbList`, `Person`) connected via `@id` cross-references into a single entity graph. Injected at prerender time, NOT runtime (AI crawlers do not execute JS). This is the single highest-leverage deliverable — the GEO audit projects it lifts Schema 8→55 and contributes ~+9 points alone.
2. **Content pages** — `/about` (RU only) and `/guides/<slug>` + `/en/guides/<slug>` (bilingual). These are normal prerendered `prerender: "full"` routes that flow through the existing `seo-routes.ts` → `entry-server.tsx` → `prerender.mjs` pipeline. The main new build-mechanics work is extending the manifest type and `EN_SIBLINGS`, not changing the pipeline.
3. **Sidecars** — `/pay` full-prerender promotion (the only billing-coupled task), `/llms.txt` + `/llms-full.txt` postbuild step, hero `fetchpriority` preload (extends `prerender.mjs`), `X-Frame-Options` + `Content-Signal` headers/robots edits.

**Primary recommendation:** Add ONE new field to `RouteMeta` (`schema?: SchemaBlock[]`) and ONE new apply helper (`applyJsonLd(html, meta)`) — that single design choice covers `Organization`, `SoftwareApplication`, `WebSite`, `Product`, `FAQPage`, `HowTo`, `BreadcrumbList`, `Person`. Schema blocks remain plain JSON literals on the route record. Resist the temptation to build a parallel schema manifest or a React-side hook — both fight the existing Phase 2/3 pattern (single manifest, single inject point, fail-fast guards).

**Confidence note:** This domain is well-trodden. Schema.org docs, Google rich-result rules, llms.txt spec, and the IETF AI-Preferences draft are stable; the only material risk is the `/pay` prerender refactor — that's a billing surface with shipped extension dependencies, so it gets its own plan task with separate verification.

### Pattern 1: Single-manifest schema injection (recommended)

**What:** Extend `RouteMeta` with `schema?: SchemaBlock[]`. Each block is a plain JS object. The `applyJsonLd` helper in `prerender.mjs` serializes them into `<script type="application/ld+json">` tags and injects before `</head>`.

**When to use:** Always — this is the chosen pattern (D-10).

**Why this over the alternatives:**
- Reuses the Phase 2 single-source-of-truth pattern (`seo-routes.ts` is already SYNC-bound to 4 files).
- Schema blocks live next to title/canonical/hreflangAlternates — easy to verify consistency in one read.
- Build-time fail-fast (`if (html === before) throw`) survives unchanged.
- Cross-references resolve correctly because both blocks live in the same JS scope (TS won't enforce them but they're easy to grep).

**Example (route record):**

### Pattern 2: `applyJsonLd` helper (mirror existing apply-chain pattern)

**Example:**

```js
// scripts/prerender.mjs (Phase 4 D-10)
function applyJsonLd(html, meta) {
  if (!meta.schema || meta.schema.length === 0) return html;
  const blocks = meta.schema.map(
    (block) => `    <script type="application/ld+json">
${JSON.stringify(block, null, 2).split("\n").map(l => "    " + l).join("\n")}
    </script>`
  ).join("\n");
  const before = html;
  // Anchor: insert before </head> (consistent with applyModulePreload — but BEFORE
  // applyMarker so applyMarker still finds its </head> anchor).
  html = html.replace("</head>", `${blocks}\n  </head>`);
  if (html === before) {
    throw new Error(`prerender(${meta.path}): no </head> anchor for JSON-LD. index.html changed?`);
  }
  return html;
}
```

**Ordering note:** `applyJsonLd` must run BEFORE `applyMarker` and `applyPaddleScript` (both also consume `</head>`). Recommended order:

```
applyMeta → applyHtmlLang → applyHreflang → applyOgLocale
  → applyJsonLd                   ← NEW
  → applyHeroPreload              ← NEW (head-only — root + /en/ only)
  → applyModulePreload
  → applySafariPreloadFallback
  → applyPaddleScript (conditional on /pay + /en/pay)
  → applyMarker
  → applyBody
```

The "consumes `</head>`" cascade is documented at `scripts/prerender.mjs:133` ("MUST be called BEFORE applyMarker"). New helpers MUST follow the same precedence rule.

### Pattern 3: Eager imports in BOTH `main.tsx` AND `entry-server.tsx`

**What:** New page components are imported synchronously in both routers. NO `React.lazy()`.

**Why:** SSR (`entry-server.tsx`) calls `renderToString()`, which cannot resolve `React.lazy()` chunks. Phase 2 hotfix `80b16be` codified this rule. `[VERIFIED: src/main.tsx:14-21 comment block]`

**Anti-pattern to avoid:** `const AboutPage = lazy(() => import("./AboutPage"))` — this would prerender an empty `<div id="root"></div>` because `renderToString` returns the Suspense fallback for unresolved lazy components.

### Pattern 4: `/pay` prerender promotion (D-12) — surgical refactor

**Trace of current state:**

### Anti-Patterns to Avoid

- **Schema injection via React `<script>` tags in JSX.** AI crawlers do NOT execute JS — schema lands too late. Phase 2 D-09 baseline.
- **`/about` added to `EN_SIBLINGS`.** D-02 forbids it. EN sibling deferred to Phase 4.1.
- **Hardcoding the schema bundle inside `prerender.mjs`.** Breaks the per-route pattern; schema lives in `seo-routes.ts`.
- **Bare `<Link>` (vs `LocalizedLink`) for new navbar/footer entries.** Would strip the `/en/` prefix on click. `CONVENTIONS.md` Anti-Convention.
- **`React.lazy()` for `AboutPage` / `GuidePage`.** SSR can't resolve. Phase 2 hotfix precedent.
- **Adding CSP in vercel.json alongside X-Frame-Options.** D-14 + PROJECT.md keeps CSP out-of-scope. Verify the planner does NOT silently bundle a CSP edit in.
- **Generating guide / about copy with an LLM.** D-07 forbids — defeats E-E-A-T position.
- **`<html lang>` mutation at runtime.** `CONVENTIONS.md` Anti-Convention; baked at build time.

## Don't Hand-Roll

| Problem | Don't build | Use instead | Why |
|---------|-------------|-------------|-----|
| JSON serialization of schema blocks | Custom string concatenation, template literals with manual escaping | `JSON.stringify(block, null, 2)` | Handles quote escaping, nested objects, null/undefined values for free. Build fails loudly if `block` has cycles. |
| HTML escaping in injected schema | Custom escaper | Reuse existing `escapeHtml` / `escapeAttr` from `prerender.mjs:27-32` | Already audited. JSON-LD `<script>` block content does not need HTML escaping — JSON is valid in `<script>` as long as it does not contain `</script>` (and a schema literal won't). |
| FAQ component | Custom accordion from scratch | Plain `<dl>` / `<details>` / Radix `Accordion` (already installed `@radix-ui/react-accordion@1.2.3`) | Reuse existing primitive; semantic + accessible. | Image preload tag generation | String template inside JSX | Reuse `applyHeroPreload(html)` head-injection in `prerender.mjs` mirroring `applyModulePreload` | Existing pattern, fail-fast guard, runs at build time. Don't import vite-imagetools output into prerender.mjs — read the rendered SSR HTML for the hero `<img src>` and emit a `<link rel="preload">` mirroring it. |
| `llms.txt` parser/writer | Custom format library | Plain Markdown via `fs.writeFile` | The llms.txt spec IS just Markdown (H1 + blockquote + H2 sections + bullet links). `[CITED: https://llmstxt.org/]` |

## Common Pitfalls

### Pitfall 1: `@id` cross-references stale or inconsistent

**What goes wrong:** `Organization.founder` references `#person-founder` but the Person block is on a different page; or `SoftwareApplication.publisher` references `#org` but two pages embed slightly different Organization blocks (different `name` or `description`). AI crawlers see two disconnected entities instead of one.

**Why it happens:** Schema literals are written per-page but the entity graph is global. Easy to introduce drift when copy-pasting.

**How to avoid:** Define `ORG_BLOCK`, `WEBSITE_BLOCK`, `PERSON_FOUNDER_BLOCK` etc. as TOP-LEVEL CONSTANTS in `seo-routes.ts` (or a sibling `scripts/schema-blocks.ts`). Each route's `schema` array references these constants. ANY page that embeds Organization embeds the SAME object — guaranteed by `===` identity.

**Warning signs:** Google Rich Results Test shows separate "Organization" entries for `/` and `/en/` with different names. Schema validator flags missing `@id` referents.

### Pitfall 2: `applyJsonLd` apply-chain ordering breaks `applyPaddleScript`

**What goes wrong:** A new helper that consumes `</head>` runs after `applyPaddleScript`, so Paddle's preconnect+script tag silently gets pushed below the new JSON-LD blocks — or worse, the script tag's `</head>` anchor disappears.

**Why it happens:** `prerender.mjs:133-140` already documents this trap: "MUST be called BEFORE applyMarker (which consumes `</head>`)". Same applies to every `</head>`-anchored helper.

**How to avoid:** Place `applyJsonLd` in the apply chain BEFORE the `</head>`-consumers (`applyModulePreload`, `applySafariPreloadFallback`, `applyPaddleScript`, `applyMarker`). Recommended position: right after `applyOgLocale`, before `applyModulePreload`.

**Warning signs:** Build succeeds but `curl /pay | grep paddle.com` returns nothing. Or `applyMarker`'s fail-fast guard throws because `</head>` was already replaced.

### Pitfall 3: `/pay` prerender promotion breaks Paddle invariant

**What goes wrong:** Promoting `/pay` to `prerender: "full"` requires `PayPage` to render via `renderToString()` in `entry-server.tsx`. If `PayPage` is still `React.lazy()` in `main.tsx`, the client-side hydration sees a mismatched tree (server rendered something, client renders Suspense fallback) — React #418/#423.

**Why it happens:** Phase 2.1 D-01 deliberately marked PayPage as lazy to keep landing-bundle small. D-12 reverses that for `/pay` only.

**How to avoid:** Either eager-import `PayPage` in BOTH routers (Option A above), or build a separate `PayPageSSR.tsx` that exports only the static block (Option B). Plan must explicitly call out which option is chosen; both must be reflected in `main.tsx` + `entry-server.tsx`.

**Warning signs:** Browser console shows React #418 / "Hydration failed" on `/pay` after deploy. Lighthouse reports a hydration crash. Paddle modal fails to open because the click handler is on the wrong node.

### Pitfall 4: `applyPaddleScript` gate logic survives the tier flip

**What goes wrong:** The current gate at `prerender.mjs:192` is `if (meta.path === "/pay" || meta.path === "/en/pay")`. If a planner accidentally rewrites this to `if (meta.prerender === "head" && (meta.path …))` to "fix" the head/full distinction, Paddle stops injecting on `/pay` after the flip to `"full"`.

**Why it happens:** Programmer trying to be clever about gating logic when promoting tiers.

**How to avoid:** Keep the path gate, change only `meta.prerender` in `seo-routes.ts`. The current gate is path-based for a reason — `applyPaddleScript` runs BEFORE `applyBody`, regardless of tier.

**Warning signs:** PayPage shows "SDK blocked" error after deploy; `window.Paddle` undefined.

### Pitfall 5: `EN_SIBLINGS` SYNC chain breaks for `/guides/<slug>`

**What goes wrong:** Adding `/guides/<slug>` to `seo-routes.ts` but not to `EN_SIBLINGS` (or vice versa). `LocalizedLink` to `/guides/<slug>` from an EN page would stay un-rewritten (wrong sibling), and `LangSwitcher` toEnTarget returns null — language flip in place, but URL never moves to `/en/guides/<slug>`.

**Why it happens:** The SYNC contract is convention-enforced, not compiler-enforced.

**How to avoid:** Add `/guides/<slug>` to the SYNC checklist: (1) `scripts/seo-routes.ts` (RU + EN entries with reciprocal hreflang), (2) `src/i18n/paths.ts` `EN_SIBLINGS`, (3) `src/main.tsx` `<Route path="/guides/:slug">` AND `<Route path="/en/guides/:slug">`, (4) `scripts/entry-server.tsx` same two routes (eager imports). Plan-check this list as a single atomic step.

**Note on dynamic segments:** `EN_SIBLINGS` is currently a Set of literal paths. For `/guides/<slug>`, you have two options:
- **Option A (simpler):** Add `"/guides/<slug>"` literal to `EN_SIBLINGS` for the one anchor guide. Phase 4.1 extends with more literals.
- **Option B (scalable):** Convert `EN_SIBLINGS` to predicate-based: `Set` of literals OR a prefix array (`/guides/`). Phase 4.1 doesn't need this until volume grows.

Recommend Option A in Phase 4 (one anchor, ship the literal); revisit in Phase 4.1.

**Warning signs:** Click `/guides/<slug>` link from `/en/` and URL becomes `/guides/<slug>` instead of `/en/guides/<slug>`. Or `npm run build` succeeds but `/en/guides/<slug>` returns SPA-fallback HTML (no per-route prerender).

### Pitfall 6: Schema "visible content" rule for `Product` on `/pay`

**What goes wrong:** Ship `Product` schema with `offers.price = "199"` but the page body (post-prerender) still renders the price via React only — Google flags "Schema doesn't match visible content" and may suppress rich results entirely.

**Why it happens:** D-12 is precisely the prevention. But if the planner ships `Product` schema in Phase 4 WITHOUT also shipping the prerender promotion, the rule is violated.

**How to avoid:** Sequence — promote `/pay` to `prerender: "full"` BEFORE (or in the same commit as) the `Product` schema injection. If `/pay` slipped to a later wave, schedule it before the schema bundle for `/pay`.

**Warning signs:** Google Search Console "Merchant listing" report shows warnings on `/pay`. Rich Results Test passes structure but flags "missing visible content".

### Pitfall 7: HowTo / FAQ rich-result deprecation makes the planner doubt the value

**What goes wrong:** Planner reads that Google has DEPRECATED HowTo rich results (Sept 2023, fully removed 2024-2025) and FAQ rich results (May 2026, removed June 2026) and concludes that shipping `HowTo` + `FAQPage` schema is pointless. The planner descopes the schema and the GEO score uplift evaporates.

**Why it happens:** The deprecation announcements are loud; the "Bingbot, PerplexityBot, RAG crawlers continue to consume the markup" caveat is buried in the footnotes. `[CITED: https://developers.google.com/search/blog/2023/08/howto-faq-changes]`

**How to avoid:** Explicit guidance in the plan: HowTo + FAQ markup is shipped for AI crawler consumption (ChatGPT, Claude, Perplexity, Bing Copilot, Gemini), NOT for Google rich results. Google has no penalty for unused markup. Phase 4 is explicitly GEO (Generative Engine Optimization), not classic SERP-rich-results SEO.

**Warning signs:** Plan-checker pushes back on shipping FAQPage schema "because Google deprecated it". Verify the rationale is captured in plan task descriptions.

### Pitfall 8: `Content-Signal` directive — wrong vocabulary

**What goes wrong:** CONTEXT.md D-14 specifies `Content-Signal: search=yes, ai-train=yes, ai-retrieval=yes`, but Cloudflare's actual three-vocabulary terms are `search`, `ai-train`, and **`ai-input`** (NOT `ai-retrieval`). `[CITED: https://contentsignals.org/, https://blog.cloudflare.com/content-signals-policy/]`

**Why it happens:** The IETF AI-Preferences vocabulary and the Cloudflare Content-Signal vocabulary use different keywords; "ai-retrieval" is intuitive but not what the spec uses.

**How to avoid:** Plan SHOULD write `Content-Signal: search=yes, ai-train=yes, ai-input=yes` (Cloudflare-compatible) — OR if the user prefers IETF's broader Content-Usage vocabulary, use that syntax instead (`Content-Usage: …`). Decision point for the planner: flag this in plan-check as a wording correction to D-14, then confirm with user before shipping the robots.txt edit.

**Warning signs:** Cloudflare's managed robots.txt parser ignores an unknown signal name. Some crawlers also surface validation warnings.

### Pitfall 9: Hero `<img>` preload mismatch with vite-imagetools `srcset`

**What goes wrong:** `applyHeroPreload` emits `<link rel="preload" as="image" href="/assets/feature-models-w400.webp">` — but the actual hero rendered by React picks a different srcset entry depending on viewport. Preload fetches an image the browser never paints, LCP regresses instead of improves.

**Why it happens:** vite-imagetools emits multi-width WebP + PNG outputs. The browser's image selection happens after preload runs.

**How to avoid:** Use `imagesrcset` + `imagesizes` on the preload tag (matching the `<img sizes=>` attribute) so the browser preloads the SAME variant it would have chosen for the responsive image. `[CITED: https://web.dev/articles/preload-responsive-images]`

```html
<link rel="preload" as="image"
      href="/assets/feature-models-w400.webp"
      imagesrcset="/assets/feature-models-w400.webp 400w, /assets/feature-models-w1100.webp 1100w"
      imagesizes="(max-width: 768px) 100vw, 50vw"
      fetchpriority="high">
```

Also: the `App.tsx:170-189` hero section doesn't actually contain a raster `<img>` — the visual is the `<OptenHeroAnimation>` SVG. The `<img>` candidates for LCP on `/` are: (a) the `highlight.svg` at line 176 (small decorative, NOT LCP), (b) `chrome-lg.svg` + `yandex-lg.svg` in `BrowserIcons` (small icons, NOT LCP). **The actual LCP element on mobile is likely the H1 text + the feature-models PNG below the fold.** Confirm via PageSpeed MCP what the actual LCP element is before preloading the wrong one.

**Warning signs:** PageSpeed LCP barely moves after preload ships, or moves the wrong direction. PageSpeed flags "Preload key requests" with a different image than the one being preloaded.

### Pitfall 10: llms-full.txt encoding pitfalls

**What goes wrong:** Concatenating 12-15 prerendered HTML bodies into `llms-full.txt` includes leftover entities (`&amp;`, `&quot;`), inline `<script type="application/ld+json">` JSON, navigation chrome (the same navbar on every page), and so on. The result is noisy and over-budget on AI-readable density.

**Why it happens:** "Strip HTML tags" looks easy but the prerendered output has lots of structural noise.

**How to avoid:** Two-pass extraction —
1. For each `dist/<route>/index.html`: parse with a lightweight regex to extract the inside of `<div id="root">…</div>`; THEN strip remaining tags; THEN decode entities; THEN collapse whitespace.
2. Skip the navbar/footer markers (start/end string contains tells, can be regex-removed) OR target the actual content section (e.g., extract everything between `<main>` and `</main>` if components add a main wrapper).

If size still exceeds 50 KB, apply the CONTEXT discretion rule: keep root + guide + about + pay only.

Concrete approach: write `scripts/llms.mjs` as ~80 lines of plain Node, no deps; if pulled to install `node-html-parser`, that's fine but not necessary.

### Example 5: applyHeroPreload (D-13)

```js
// scripts/prerender.mjs (Phase 4 D-13)
// Inject hero image preload into root pages only ("/", "/en/"). Other prerendered
// pages have either no above-the-fold raster image, or use the OptenHeroAnimation
// SVG (not subject to LCP preload optimization).
function applyHeroPreload(html, meta) {
  // Only / and /en/ render the hero section. Other routes return early.
  if (meta.path !== "/" && meta.path !== "/en/") return html;

  // Hero image candidates depend on viewport — vite-imagetools emits a srcset.
  // The actual LCP element on mobile is likely below-the-fold; preload the most
  // common LCP candidate identified via PageSpeed MCP analysis before shipping.
  // Pitfall 9: use imagesrcset + imagesizes to match the <img> srcset exactly.
  const tag = `    <link rel="preload" as="image"
      href="/assets/landing-design/feature-models-w400.webp"
      imagesrcset="/assets/landing-design/feature-models-w400.webp 400w, /assets/landing-design/feature-models-w1100.webp 1100w"
      imagesizes="(max-width: 768px) 100vw, 50vw"
      fetchpriority="high" />`;

  const before = html;
  html = html.replace("</head>", `${tag}\n  </head>`);
  if (html === before) {
    throw new Error(`prerender(${meta.path}): no </head> anchor for hero preload.`);
  }
  return html;
}
```

**Pre-implementation gate:** Before committing to `feature-models` as the preload target, run PageSpeed MCP on production `/` and `/en/` to identify the actual LCP element. Phase 2.2 already removed `fetchPriority` from `BrowserIcons` (`App.tsx:75` comment) — that was a different image entirely. The LCP element may be the H1 text (in which case preload the Unbounded font, which is ALREADY in `index.html:96`), not an image at all.

```js
// Pitfall 10: extract <div id="root"> content only, skip <script>/<style>.
function htmlToPlainText(html) {
  return html
    .replace(/<script[\s\S]*?<\/script>/g, "")
    .replace(/<style[\s\S]*?<\/style>/g, "")
    .replace(/<[^>]+>/g, " ")
    .replace(/&amp;/g, "&").replace(/&quot;/g, '"').replace(/&lt;/g, "<").replace(/&gt;/g, ">").replace(/&nbsp;/g, " ")
    .replace(/\s+/g, " ")
    .trim();
}

const fullSegments = [];
for (const meta of indexed) {
  const file = meta.path === "/"
    ? resolve(DIST, "index.html")
    : resolve(DIST, meta.path.replace(/^\//, ""), "index.html");
  try {
    const html = await readFile(file, "utf-8");
    const bodyMatch = html.match(/<div id="root">([\s\S]*?)<\/div>\s*<script/);
    const body = bodyMatch ? bodyMatch[1] : "";
    fullSegments.push(`\n\n# ${meta.title}\nURL: ${meta.canonical}\n\n${htmlToPlainText(body)}`);
  } catch {
    // route was emitted at "head" tier — skip
  }
}
let fullBody = `# Opten — Full Content\n\nGenerated ${new Date().toISOString().split("T")[0]} from prerendered HTML.\n${fullSegments.join("")}`;

// Size cap heuristic per CONTEXT (~50 KB)
const SIZE_CAP = 50 * 1024;
if (fullBody.length > SIZE_CAP) {
  // Truncate to priority routes only: root + about + guide + pay
  const priorityPaths = ["/", "/en/", "/about", `/guides/${process.env.GUIDE_SLUG ?? ""}`, "/pay"];
  const priorityRoutes = indexed.filter(r => priorityPaths.includes(r.path));
  // (re-emit with the smaller set)
}
await writeFile(resolve(DIST, "llms-full.txt"), fullBody, "utf-8");

console.log(`✓ llms.txt + llms-full.txt emitted (${fullBody.length} bytes)`);
```

### Q1: Hero LCP element identification — what's actually the LCP target on `/`?

- **What we know:** PageSpeed Mobile reported LCP 3.3 s pre-Phase-4 baseline (GEO audit §High-7). `App.tsx:170-189` Hero section renders text + small `highlight.svg` + browser icons + `<OptenHeroAnimation>` (SVG) + InstallButton.

### Q2: `Content-Signal` vs `Content-Usage` vs vocabulary choice (Pitfall 8)

- **What we know:** Cloudflare's `Content-Signal` uses `search`, `ai-train`, `ai-input`. The IETF AI-Preferences draft (`draft-ietf-aipref-attach-04`) defines `Content-Usage` with a larger vocabulary.
- **What's unclear:** CONTEXT D-14 specifies `ai-retrieval` — not in either spec.
- **Recommendation:** Plan should write `Content-Signal: search=yes, ai-train=yes, ai-input=yes` (Cloudflare-aligned, widely-deployed) and flag the wording correction to the user during discuss-phase if not yet confirmed. Optional: also emit `Content-Usage: search=yes, ai-train=yes` to align with IETF — both directives can coexist in one robots.txt.

### Q3: Founder photo source (D-03)

- **What we know:** D-03 requires founder photo for Person schema `image` property.
- **What's unclear:** Source of the photo file — does the user have one ready? File format / dimensions / where to store in `public/`?
- **Recommendation:** Planner asks the user during plan-check to supply the file; store at `public/about/founder-photo.jpg` (or `.webp`). Person schema references absolute URL `https://opten.space/about/founder-photo.jpg`. Aspect ratio: square or 4:3 works for both schema and visual page placement.

### Q4: Anchor guide topic confirmation (D-04)

- **What we know:** Default is "How to write prompts for Midjourney v7"; alternatives considered Flux / SD 3.5 if Telegram-chat / CWS-review topic frequency suggests otherwise.
- **What's unclear:** Has the user sampled the source pool yet? If not, the planner needs to do it (or punt to user).

### Q5: About-link visibility on EN navbar (D-16)

- **What we know:** `/about` is RU-only; EN navbar entry has no destination.
- **What's unclear:** Hide entirely vs render a "Coming soon in EN" stub link to `/about` (where the user lands on a Russian page).
- **Recommendation:** **Hide entirely.** Showing a link that leads to RU content on an EN page is worse UX than the link being absent. ### Q6: Reciprocal hreflang for `/about` (D-02 edge case)

- **What we know:** D-02 says `/about` has hreflang `ru` self-canonical only — no `en` alternate.
- **What's unclear:** The current `applyHreflang` ALWAYS emits a triplet (`ru`, `en`, `x-default`) from `meta.hreflangAlternates`. For `/about` with no EN sibling, what should `meta.hreflangAlternates.en` be?
- **Recommendation:** Either (a) change `applyHreflang` to skip the `en` line when `meta.hreflangAlternates.en === null`, or (b) set `meta.hreflangAlternates = { ru: ".../about", en: ".../about", xDefault: ".../about" }` — all three point to the RU canonical, signaling "this content is not bilingual". Google docs prefer option (a) for unambiguous signaling. Plan-task should pick one and document; option (a) is the cleaner intent.
