# Phase 2.1 — Validation Strategy

> Per-phase validation contract for feedback sampling during execution.

This phase has **no automated test framework** in the project (CLAUDE.md "no tests, no ESLint config, no `typecheck` script — TS errors surface during `vite build`"). Validation is **build-driven + manual smoke checks**: every change must keep `npm run build` green, and acceptance criteria are verified via Lighthouse mobile + manual mobile-tap-latency tests.

---

## Test Infrastructure

| Property | Value |
|----------|-------|
| **Framework** | none (build + manual smoke; no Jest/Vitest/Playwright in repo) |
| **Config file** | none — `vite build` IS the type-and-bundle check |
| **Quick run command** | `npm run build` (exits non-zero on TS errors) |
| **Full suite command** | `npm run build && npx unlighthouse --site http://localhost:8080` (after `npx http-server dist`) |
| **Estimated runtime** | ~30s build + ~60s Lighthouse run |

**Wave 0 install (optional, dev-only, deferred unless researcher recommends):**
- `npm install --save-dev rollup-plugin-visualizer` (treemap generator for bundle-size acceptance check)
- `npm install --save-dev unlighthouse` OR use `npx unlighthouse` per-run

---

## Sampling Rate

- **After every task commit:** Run `npm run build` (≤ 30s; non-zero exit = task is red).
- **After every plan wave:** Run `npm run build`, inspect `dist/index.html` for `<link rel="modulepreload">`, inspect `dist/assets/index-*.js` size, run `npx serve dist` + manual tap test in Chrome DevTools mobile emulation.
- **Before `/gsd-verify-work`:** Full Lighthouse mobile run on production preview. Confirm Performance ≥ 90 (or document why ≥ 75 is acceptable per RESEARCH.md finding that 220 KB target unrealistic), CLS = 0.
- **Max feedback latency:** 30s per task commit, 90s per wave.

## Success Criteria (mapped from ROADMAP.md + RESEARCH.md revisions)

- **SC-1 (treemap):** Main chunk for `/` does NOT contain `chrome.runtime` or `Paddle` references. Treemap (`dist/stats.html` from rollup-plugin-visualizer) shows PayPage / AccountPage / SuccessPage / DownloadSkillPage as separate chunks. **Build-driven.**
- **SC-2 (Lighthouse):** Mobile Performance score ≥ 75 on production (revised from ROADMAP's ≥ 90 per RESEARCH.md ceiling analysis — React + Radix/MUI/motion floor is ~280 KB). **Manual + Lighthouse.**
- **SC-3 (bundle reduction):** Main chunk for `/` reduced by ≥ 10% from baseline 362 KB (revised from ROADMAP's ≥ 40% per RESEARCH.md). Concrete target: 362 KB → ≤ 325 KB. **Build-driven.**
- **SC-4 (CLS):** Cumulative Layout Shift = 0 on `/` per Lighthouse mobile. Achieved via D-04 (width/height attributes). **Lighthouse.**
- **SC-5 (tap latency):** Navbar menu hamburger registers `setOpen` within 500ms of FCP on simulated Slow 4G + 4× CPU throttling. **Manual via Chrome DevTools.**
- **SC-6 (no regression):** All 9 routes still respond. Prerendered routes still have distinct titles/meta/canonical. Paddle modal still opens on `/pay`. `__PRERENDER_PATH` discriminator unchanged. OG card unfurl on Telegram still works. **Mixed: build + manual.**

**Revision note:** Per RESEARCH.md "Achievable Bundle Reduction" section, the ROADMAP's ≥ 40% target was set without bundle treemap data; researcher's empirical run shows ~10% is the realistic ceiling. Phase 2.1 acceptance uses revised targets (SC-2 ≥ 75, SC-3 ≥ 10%) — planner and verifier MUST honor the revised numbers, not the ROADMAP draft.
