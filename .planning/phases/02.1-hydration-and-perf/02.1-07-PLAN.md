<objective>
Switch the four SPA-fallback route imports in `src/main.tsx` from static to `React.lazy()`, and wrap `<Routes>` in a single `<Suspense fallback={<RouteLoading />}>` boundary. This is the **highest-risk plan in Phase 2.1** â€” it directly touches the hydration detector's neighborhood and the route tree that `scripts/entry-server.tsx` mirrors. It runs LAST per Phase 2's lesson (the `80b16be` hotfix proved that touching `main.tsx` carries SSR-mismatch risk).

**Lazy targets (per CONTEXT.md D-01 and RESEARCH.md Suspense safety analysis):**
- `PayPage` â€” heavy: Paddle wrapper + Radix Select + currency logic (~19 KB raw chunk per RESEARCH.md).
- `SuccessPage` â€” light but currently bundled with Paddle-dependent code (~2.7 KB chunk).
- `AccountPage` â€” heavy: chrome.runtime flow + subscription UI (~13 KB chunk).
- `DownloadSkillPage` â€” heavy: JWT + Pro-gating (~3.5 KB chunk).

**MUST remain static (PATTERNS.md "Critical Invariants" #4 and RESEARCH.md "Suspense + hydrateRoot at Route Level"):**
- `App` â€” rendered for `/` by `scripts/entry-server.tsx`.
- `WelcomePage` â€” rendered for `/welcome` by entry-server.
- `PrivacyPage`, `TermsPage`, `RefundPage` â€” rendered for `/privacy`, `/terms`, `/refund` by entry-server.

**Why Suspense + hydrateRoot is safe here (RESEARCH.md lines 329â€“363):**
The 5 prerendered routes are static imports â€” their components are resolved synchronously before React renders, so `<Suspense>` never enters fallback during hydration for these routes. The 4 SPA-fallback routes never run through `hydrateRoot` (the `__PRERENDER_PATH` detector wipes `#root` and uses `createRoot` for them) â€” so `<Suspense>` fallback only activates on the createRoot path, which has no SSR tree to reconcile against. No hydration mismatch possible.
