## Executive Summary

Phase 2.1 is a mechanical bundle optimization that is safe to execute. All four scope areas (code-split, modulepreload, image dimensions, PNG→WebP) have been validated against the actual codebase. The main chunk drops from **362 KB to ~325 KB** after code-splitting the four SPA-fallback routes — a 10.2% raw reduction or ~6.7 KB gzip saving (105 KB → ~98.5 KB gzip). The ROADMAP target of "≤ 220 KB" is unrealistic for this tech stack: React + react-dom + react-router + all the Radix/MUI dependencies cannot be compressed below ~280 KB without removing packages, which is out of scope. The realistic and valuable target is ≤ 325 KB raw / ≤ 99 KB gzip on the landing-critical path.

**Critical version finding (BLOCKER):** `vite-imagetools` latest (10.0.0) requires Vite ≥ 7.0.0. This project is on Vite 6.3.5. The last Vite 6-compatible version is **`vite-imagetools@6.2.9`**, which installs and passes both the client and SSR builds cleanly. D-05 must pin `vite-imagetools@6.2.9`.

**Dynamic-src finding:** `vite-imagetools` requires statically-analyzable import paths. The `feature-interface-${suffix}.png` and `feature-enhance-${suffix}.png` patterns (where `suffix` is `"ru"` or `"en"` at runtime) CANNOT be processed via query-string imports. Both language variants must be imported statically and switched in JSX. All other images in `src/app/App.tsx` use static paths and are straightforward.

**D-03 finding:** Vite 6 does NOT emit `<link rel="modulepreload">` for the entry chunk in the default `dist/index.html`. It emits `<script type="module" crossorigin src="/assets/index-HASH.js">` directly. `prerender.mjs` must synthesize the `<link rel="modulepreload">` tag from the `<script>` tag's `src` attribute.

Total estimated LOC changes: ~200 lines across 6 files (main.tsx, vite.config.ts, prerender.mjs, App.tsx, plus new RouteLoading.tsx). No structural changes to the Phase 2 invariants.

**Primary recommendation:** Proceed with all four D-01..D-07 tasks as planned, pinning `vite-imagetools@6.2.9`. The dynamic image path constraint (two feature card images + one steps image) adds ~20 lines of import wiring vs static paths, but does not change the architecture.

## Vite modulePreload Default Behavior (D-03)

**Verified by:** reading `dist/index.html` after `npx vite build`. [VERIFIED: local build output]

**Finding:** Vite 6.3.5 does NOT emit `<link rel="modulepreload">` for the entry chunk under default configuration. The built `dist/index.html` contains:

```html
<script type="module" crossorigin src="/assets/index-nE1Iwdmv.js"></script>
<link rel="stylesheet" crossorigin href="/assets/index-B8zfXOnQ.css">
```

There is no `<link rel="modulepreload">` tag. This is confirmed: `build.modulePreload` defaults to `{ polyfill: true }` in Vite 6, which injects a `modulepreload` polyfill but does NOT add `<link rel="modulepreload">` tags for the entry chunk itself.

**Consequence for prerender.mjs (D-03):** The script must SYNTHESIZE the `<link rel="modulepreload">` tag. The synthesis path is:

1. Read `dist/index.html` (the template that `prerender.mjs` already reads as `template`).
2. Extract the entry JS chunk `src` from the `<script type="module" crossorigin src="...">` tag using a regex: `/src="(\/assets\/index-[^"]+\.js)"/`.
3. Insert `<link rel="modulepreload" href="${chunkSrc}">` as the FIRST element after `<head>`, BEFORE the Paddle CDN script tag.

**Ordering constraint (from INTEGRATION-CONTRACT.md §6):** The `<link rel="modulepreload">` must be placed AFTER the Paddle `<script src="https://cdn.paddle.com/paddle/v2/paddle.js">` tag. The Paddle script must remain synchronous and execute BEFORE `main.tsx` runs. `modulepreload` only hints the browser to fetch in parallel — it does not change execution order. Placing `modulepreload` after Paddle's `<script>` is safe and preserves the Paddle init contract.

**Recommended injection point in `applyModulePreload(html, preloadHref)`:**
```js
// Insert before </head>, after existing tags
return html.replace("</head>", `    <link rel="modulepreload" href="${preloadHref}">\n  </head>`);
```

**`crossorigin` attribute:** Omit. `opten.space` serves assets from the same origin (Vercel). The `crossorigin` attribute on `modulepreload` is only required when assets are served from a different origin with CORS. [VERIFIED: dist/index.html Vite output uses `crossorigin` on the `<script type="module">` itself but that is a different purpose]

### Dynamic import path limitation

`vite-imagetools` requires **statically-analyzable import paths** (compile-time known). Dynamic template literals like `src={`${ASSET_ROOT}/feature-interface-${suffix}.png`}` are NOT processable by the plugin — Vite's dynamic import analysis drops the query-string parameters, and `vite-imagetools` never sees the file.

**Affected image paths in `src/app/App.tsx`:**

| Component | Current dynamic pattern | Constraint |
|-----------|------------------------|------------|
| `FeatureCards` (card 2) | `` `${ASSET_ROOT}/feature-interface-${suffix}.png` `` | suffix = "ru" or "en" |
| `FeatureCards` (card 4) | `` `${ASSET_ROOT}/feature-enhance-${suffix}.png` `` | suffix = "ru" or "en" |
| `Steps` | `` `${ASSET_ROOT}/steps-inner-${lang}.png` `` | lang = "ru" or "en" |

**Also affected (partner logos):** `PartnerLogo` uses `` `${ASSET_ROOT_PARTNERS}/${name}.png` `` where name is a prop (`"syntx" | "freepik" | "higgsfield" | "canva"`). Same constraint.

**Resolution pattern — hand-import both variants and switch in JSX:**

```tsx
// Static imports at module level (vite-imagetools processes these at build time)
import featureInterfaceRuSrc from '/assets/landing-design/feature-interface-ru.png?w=1100&format=webp;png&as=picture'
import featureInterfaceEnSrc from '/assets/landing-design/feature-interface-en.png?w=1100&format=webp;png&as=picture'

// In component:
const featureInterfaceSrc = lang === "ru" ? featureInterfaceRuSrc : featureInterfaceEnSrc;
// Then render as <Picture data={featureInterfaceSrc} width={1100} height={424} alt="" className="..." />
```

**Recommendation for "Claude's Discretion" on `as=picture` vs `as=metadata`:**

Use **`as=picture`** mode. It returns a `PictureSource` object containing `sources` array and `img` properties that map directly to `<picture><source><img></picture>` markup. This allows a thin `<Picture>` wrapper component (~20 lines) that handles all images uniformly. The `as=metadata` mode returns dimensions and srcset strings — more flexible but requires hand-rolling the `<picture>` element every call site. Since every image in this phase follows the same WebP+PNG pattern, `as=picture` is cleaner.

**Note on vite-imagetools@6.2.9 `as=picture` syntax:** In v6.x, the `as=picture` directive is supported. The query string syntax is `?w=1100&format=webp;png&as=picture`. The result is a `Picture` object with `{ sources: [{ srcset, type }], img: { src, width, height, ...} }`. [ASSUMED — based on v6.x changelog and README; verify with a single test import during Wave 0]

## Suspense + hydrateRoot at Route Level (D-01 Verification)

**Finding: SAFE for this codebase's specific hydration architecture.** [VERIFIED: reading main.tsx hydration detector logic]

The concern about Suspense + hydrateRoot mismatch applies when:
1. The SSR output contains a `<Suspense>` boundary that was resolved at render time
2. The client wraps the same subtree in `<Suspense>` but the lazy component hasn't loaded yet
3. React sees a mismatch between "resolved subtree" (server) vs "Suspense fallback spinner" (client)

**This project's architecture avoids the mismatch entirely:**

The `main.tsx` hydration detector (Phase 2 D-06) routes traffic at the root level:

```ts
if (root.hasChildNodes() && pathMatches) {
  hydrateRoot(root, tree);   // prerendered routes: /, /welcome, /privacy, /terms, /refund
} else {
  createRoot(root).render(tree);  // SPA-fallback routes: /pay, /account, /success, /dashboard/*
}
```

**For `hydrateRoot` path (prerendered routes):**
- These routes' components (`App`, `WelcomePage`, etc.) are STATIC imports — they are resolved synchronously before React renders.
- The `<Suspense>` boundary wraps `<Routes>`. When React encounters the `<Route path="/" element={<App />}>` match, `<App>` is already loaded (static import). There is no suspended component in the hydration tree.
- React does NOT enter the Suspense fallback during hydration — the lazy components (`PayPage` etc.) are simply not rendered for the `"/"` route match.
- **Result: no hydration mismatch.** The client tree and server tree are structurally identical (modulo lazy route definitions that are never evaluated for these routes).

**For `createRoot` path (SPA-fallback routes):**
- These routes receive the landing page's prerendered HTML via Vercel's SPA rewrite. The `pathMatches` check in main.tsx detects the mismatch and routes to `createRoot` (wiping `#root` first).
- `createRoot` does NOT hydrate — it renders from scratch. The `<Suspense>` boundary activates normally, showing `<RouteLoading>` briefly while the lazy chunk fetches.
- **Result: no hydration mismatch.** `createRoot` never reconciles against existing DOM.

**Supporting evidence:** React 18's reconciler documentation confirms that `<Suspense>` boundaries around subtrees that are NOT suspended during hydration pass through without triggering fallbacks or mismatches. The constraint that caused Phase 2 hotfix `80b16be` was different: it was `renderToString` (server-side) encountering a `<Suspense>` boundary with a `React.lazy()` component — `renderToString` cannot resolve dynamic `import()` and throws. Phase 2.1's lazy imports are ONLY in `main.tsx` (client entry), never in `entry-server.tsx`. [VERIFIED: entry-server.tsx does not import PayPage, AccountPage, SuccessPage, or DownloadSkillPage]

**Conclusion:** The `<Suspense>` wrapper in `main.tsx` is safe. The hydration-mismatch risk is zero for this architecture.

## Common Pitfalls

### Pitfall 1: Installing vite-imagetools@latest (10.x) against Vite 6

**What goes wrong:** `npm install vite-imagetools` installs 10.0.0, which peer-requires `vite@>=7.0.0`. The build fails with: `Error: Vite 7+ required`.
**Why it happens:** The npm registry resolves `@latest` to 10.0.0.
**How to avoid:** Always pin: `npm install --save-dev vite-imagetools@6.2.9`.
**Warning signs:** `npm install` prints ERESOLVE with `peer vite@">=7.0.0"`.

### Pitfall 2: Using dynamic template literal paths with vite-imagetools

**What goes wrong:** `<img src={`${ASSET_ROOT}/feature-interface-${suffix}.png?w=1100&format=webp`}>` — the query string is dropped at runtime; the browser loads the raw PNG at full size.
**Why it happens:** `vite-imagetools` is a build-time transform. Dynamic import paths are not statically analyzable, so the transform never fires.
**How to avoid:** Import each language variant as a separate static module-level import. Switch in JSX.
**Warning signs:** Build completes without error, but the feature card images are still PNG (not WebP) in DevTools Network tab.

### Pitfall 3: Placing `<link rel="modulepreload">` BEFORE the Paddle `<script>` tag

**What goes wrong:** If `prerender.mjs` injects `<link rel="modulepreload">` as the FIRST element in `<head>` (before Paddle), the page still works — but it's cosmetically wrong and could interfere with some browser fetch priority heuristics.
**Why it happens:** Developer adds the `<link>` at `</head>` which is fine, or adds it at `<head>` which prepends.
**How to avoid:** Use `html.replace("</head>", ...)` to append before `</head>`. The Paddle `<script>` tag is in the source `index.html` and is preserved throughout prerender.mjs processing.
**Warning signs:** Check `dist/index.html` after build — Paddle `<script>` must appear before `<link rel="modulepreload">`.

### Pitfall 4: Lazy-loading App.tsx or any prerendered route

**What goes wrong:** `const App = React.lazy(() => import('./app/App.tsx'))` causes Phase 2 hotfix `80b16be` to regress — `renderToString` encounters a `<Suspense>` boundary with an unresolved `import()` and throws `React Error #419`.
**Why it happens:** `entry-server.tsx` imports `App` via static import today. If the source file changes to a lazy import pattern but `entry-server.tsx` isn't updated, the SSR build will fail. If `entry-server.tsx` IS updated with `React.lazy()` and `renderToString` runs, it throws.
**How to avoid:** Only add `React.lazy()` calls to `main.tsx` (client entry). `entry-server.tsx` must never import lazy components. The presence of `PayPage`, `AccountPage`, `SuccessPage`, `DownloadSkillPage` in neither `entry-server.tsx` nor `scripts/seo-routes.ts` already protects against this.
**Warning signs:** `entry-server.tsx` imports PayPage/AccountPage = bug. `renderToString` throws with message about unsupported Suspense during synchronous rendering.

### Pitfall 5: Forgetting `__PRERENDER_PATH` marker ordering in prerender.mjs

**What goes wrong:** The `applyModulePreload` function inserts AFTER `</head>`. The `applyMarker` function also inserts before `</head>`. If both are called in the wrong order, or one overwrites the other, the marker may be inserted twice or lost.
**Why it happens:** Multiple regex replacements on `</head>` can conflict.
**How to avoid:** `applyMarker` already inserts `${marker}\n  </head>` — replacing `</head>`. After `applyMarker`, the string `</head>` no longer exists; subsequent `applyModulePreload` calling `html.replace("</head>", ...)` will find nothing. **Run `applyModulePreload` BEFORE `applyMarker` in the loop.** The ordering should be: `applyMeta → applyModulePreload → applyMarker → applyBody`.
**Warning signs:** `grep "modulepreload" dist/index.html` returns nothing after a build that should have injected it.

### Pitfall 6: `as=picture` import type in TypeScript — needs declaration

**What goes wrong:** TypeScript throws `Cannot find module '*.png?w=1100&format=webp;png&as=picture'` because there's no type declaration for the `as=picture` query result.
**Why it happens:** `vite-imagetools` v6 ships type declarations in `node_modules/vite-imagetools/types/index.d.ts`. If the `tsconfig.json` doesn't include `types: ["vite-imagetools"]` or the global import declaration is missing, TypeScript doesn't know the return type.
**How to avoid:** Add to `src/vite-env.d.ts` or a new `src/imagetools.d.ts`:
```ts
/// <reference types="vite-imagetools" />
```
Or use the explicit `Picture` type from `vite-imagetools`:
```ts
import type { Picture } from 'vite-imagetools'
```
**Warning signs:** `vite build` fails with TypeScript errors on imagetools import lines.

### modulepreload injection in prerender.mjs (D-03)

```js
// Source: derived from verified dist/index.html structure
function applyModulePreload(html) {
  // Extract main chunk href from Vite-emitted <script type="module">
  const match = html.match(/src="(\/assets\/index-[^"]+\.js)"/);
  if (!match) return html; // no match = dev build or SSR context, skip
  const chunkHref = match[1];
  const tag = `    <link rel="modulepreload" href="${chunkHref}">`;
  return html.replace("</head>", `${tag}\n  </head>`);
}

// In the route loop, call BEFORE applyMarker:
let html = applyMeta(template, meta);
html = applyModulePreload(html);      // ← new
html = applyMarker(html, meta.path);  // ← existing (replaces </head>)
if (meta.prerender === "full") {
  html = applyBody(html, renderRoute(meta.path));
}
```
