# Phase 2.1: Hydration speedup & perceived-load optimization — Context

**Gathered:** 2026-05-15
**Status:** Ready for planning
**Mode:** `--auto` (recommended option chosen for every gray area; see `02.1-DISCUSSION-LOG.md`)

<domain>
## Phase Boundary

Close the hydration gap that Phase 2 exposed: the landing page now paints fully dark + content-complete from the first browser paint (thanks to per-route prerender), but `<button onClick>` handlers stay dead for 1-3 seconds on mobile while the 362 KB JS bundle downloads, parses, and hydrates. The user-visible failure: tapping the navbar menu hamburger / lang switcher / Paddle CTA does nothing until hydration finishes. Lower-viewport buttons appear to "work" only because by the time the user scrolled to them, hydration already completed. This phase shrinks the JS bundle on the critical path, makes the browser fetch it in parallel with HTML, and kills the secondary perceived-load issues (FOUC, layout shift from unsized images, oversized PNG payloads).

**Explicitly in this phase:**
- Route-level `React.lazy()` + `<Suspense>` for SPA-fallback routes (`/pay`, `/account`, `/success`, `/dashboard/download-skill`) — these run through `createRoot` on the client, so lazy boundaries are SSR-safe (they were never in the prerender tree).
- `<link rel="modulepreload">` for the main JS chunk in every emitted HTML (`index.html` + `dist/{route}/index.html` via `scripts/prerender.mjs`).
- Explicit `width` + `height` attributes (or CSS `aspect-ratio`) on every `<img>` in `src/app/` (~26 tags by `grep` — partner logos, feature card images, step illustrations, hero highlight, browser icons, inline SVG icons).
- PNG → WebP conversion (AVIF deferred — see D-06) for partner logos (`/assets/partners/*.png`) and feature card images (`/assets/landing-design/feature-*.png` and `steps-inner-*.png`) via `vite-imagetools` plugin with `<picture>`-based srcset + PNG fallback.

**Explicitly NOT in this phase:**
- **Lazy-loading prerendered routes** (`/`, `/welcome`, `/privacy`, `/terms`, `/refund`). React 18.3's `renderToString` (used by `scripts/entry-server.tsx`) cannot resolve `<Suspense>` boundaries — Phase 2 hotfix `80b16be` already removed Suspense from `OptenHeroAnimation` for this exact reason. Lazy-loading these routes would require switching to `renderToPipeableStream` or preloading every lazy chunk before hydration, both larger changes than this phase needs.
- **Splitting the landing tree itself** (Hero / Steps / FeatureCards / Pricing as separate chunks). Same SSR-Suspense constraint applies — would break the per-route prerender invariant locked in Phase 2 (`02-CONTEXT.md` D-06).
- **SPA-fallback hydration gap fix for `/pay` /`/account` etc.** Those routes receive the LANDING prerendered HTML via Vercel's `((?!api/).*) → /index.html` rewrite, then `main.tsx` wipes `#root` and `createRoot`-renders the correct page. The wipe+render gap is a separate problem; fixing it would require head-only prerender + skeleton body per locked-route, which touches the Paddle init contract (DEC-integration-contract-paddle-init). Deferred to a later phase if user still reports the issue after Phase 2.1 ships.
- **Welcome screenshot optimization** (`public/assets/welcome-*.png` at 500-700 KB each). `/welcome` is hit once on first install and not in the hot path for AI citation or conversion; deferred.
- **AVIF encoding.** WebP alone hits ~70-80% size reduction vs PNG and has universal mobile support (iOS 14+, Android Chrome since forever). AVIF adds another 10-15% but the iOS Safari < 16 decoder is shaky and `vite-imagetools` AVIF support is opt-in. Reassess after Phase 2.1 ships.
- **Font preloading / Google Fonts → local PT Root UI swap.** That's a separate perf issue (FOUT for PT Root UI from `fonts.googleapis.com`); deferred to a follow-up if Lighthouse mobile score still misses ≥ 90 after image + JS work.
- **Service Worker / app-shell caching.** Out of scope.

### D-01. Code-split scope: SPA-fallback routes only (NOT prerendered routes)
**Decision:** Convert these imports in [src/main.tsx](../../../src/main.tsx) to `React.lazy()` with a single shared `<Suspense fallback={<RouteLoading />}>` wrapping `<Routes>`:
- `PayPage` (heavy: Paddle wrapper + Radix UI Select + currency-toggle logic)
- `AccountPage` (heavy: chrome.runtime.sendMessage flow + subscription state UI)
- `SuccessPage` (light, but bundled alongside Paddle-dependent code in current graph)
- `DownloadSkillPage` (heavy: JWT + Pro-gating + chrome.runtime.sendMessage flow)

Keep these as STATIC imports (must hydrate synchronously from prerendered HTML — `scripts/entry-server.tsx` includes them in `renderToString`):
- `App` (landing, prerendered for `/`)
- `WelcomePage`, `PrivacyPage`, `TermsPage`, `RefundPage` (all prerendered)
- `LangProvider`, `BrowserRouter`, `Routes`, `Route` (router skeleton)

**Rationale:** Phase 2 hotfix `80b16be` proved that `renderToString` cannot resolve `<Suspense>` — eager-importing `OptenHeroAnimation` fixed React #419 on the landing. Same constraint applies to the 5 fully-prerendered routes. The 4 SPA-fallback routes go through the `createRoot` fallback path in `main.tsx` (see Phase 2 `02-CONTEXT.md` D-06), so their lazy boundaries are evaluated client-side only and produce no SSR mismatch. Estimated bundle delta: 362 KB → ~210-240 KB on `/` (Paddle + chrome.runtime code moves to separate chunks).

**Researcher to validate:** that PayPage / AccountPage / SuccessPage / DownloadSkillPage do not share any module that App.tsx also pulls in — if they do, Vite's chunk splitter will put the shared module in a shared chunk and the main-route saving will be smaller than expected. Run `npm install --save-dev rollup-plugin-visualizer`, add it to `vite.config.ts`, and produce a treemap before merging.

### D-02. Suspense fallback: minimal full-viewport dark spinner, NOT skeleton
**Decision:** A new component `src/app/components/RouteLoading.tsx` that renders a centered spinner on `#011417` background, full viewport (`min-h-screen flex items-center justify-center bg-[#011417]`). Used as the single `Suspense.fallback` wrapping the `<Routes>` in `main.tsx`. No per-route skeleton — each lazy route has a different layout and per-route skeletons triple the code-split benefit (every skeleton has to ship in the main chunk).

**Rationale:** Lazy fallback only shows for the 4 SPA-fallback routes, all of which already paint a brief loading state under the current architecture (extension users land on `/pay` from popup → see dark hero of landing → JS wipes → see PayPage). The new behavior is consistent: dark page + spinner. Avoids reinventing skeleton design per route and keeps the main chunk lean. Spinner asset should be a CSS-only `@keyframes spin` on a Tailwind-styled `div` — no SVG, no library, ~200 bytes.

### D-03. modulepreload injection: `scripts/prerender.mjs` extends per-route HTML head
**Decision:** Extend `scripts/prerender.mjs` to inject `<link rel="modulepreload" href="/assets/index-{hash}.js" />` into the `<head>` of every emitted HTML file (both `dist/index.html` and `dist/{route}/index.html`). The hash comes from Vite's manifest (`dist/.vite/manifest.json` if `build.manifest = true` in `vite.config.ts`, or read from the existing `<script type="module" src="/assets/index-XXXX.js">` tag in the source `dist/index.html` produced by `vite build`).

For the lazy chunks (D-01), DO NOT inject `modulepreload` for them — that would defeat the code-split point (preloading them on the landing means downloading them anyway). Vite will inject the right chunk references when the lazy `import()` resolves.

**Rationale:** Without modulepreload, the browser starts fetching the main JS chunk only AFTER parsing the `<script type="module">` tag in HTML, which is at end-of-body (or wherever Vite injects it). With modulepreload as the FIRST tag in `<head>`, the fetch starts in parallel with HTML download — saves 1-2 RTT on mobile networks. Vite officially documents this pattern; we just need to ensure prerender.mjs preserves it across all 6 emitted HTML files.

**Researcher to validate:** whether Vite 6's default build already emits `<link rel="modulepreload">` for the entry chunk (some configurations do, some don't). If yes, prerender.mjs just needs to preserve the tag; if no, we synthesize it from the script tag's href.

### D-04. `<img>` dimensions: `width` + `height` HTML attributes, NOT CSS aspect-ratio
**Decision:** Add `width="N"` and `height="N"` HTML attributes to every `<img>` tag in `src/app/`. Values derived from the CSS Tailwind classes already on each image:
- Logo `h-[18px]` → `width="X" height="18"` (compute X from logo SVG aspect ratio: 84×18 = ratio 4.67, so width=84)
- Partner logos `h-[40px] w-[134px]` → `width="134" height="40"`
- BrowserIcons `size-[40px]` → `width="40" height="40"` (per icon)
- Step inner image `max-w-[430px]` → `width="430" height="N"` where N comes from the PNG's native aspect ratio
- Feature card images (`w-[calc(100%-64px)]`) → need a sentinel `width` + `height` based on PNG native size; the CSS will override the rendered size but the attributes give the browser the aspect ratio to reserve space.
- Inline icons `size-7` → `width="28" height="28"`
- Cross.svg `w-[150px] md:w-[205px]` → `width="150" height="N"` from native SVG viewBox (mobile-first attribute; CSS scales on desktop)

**Rationale:** HTML width/height attributes are used by the browser to compute aspect ratio BEFORE CSS loads (CSS Working Group's `aspect-ratio-from-attributes` algorithm, shipped in Chrome 79, Safari 14, Firefox 71). CSS `aspect-ratio` works but only after the stylesheet parses — defeats the point of using it for CLS prevention. Setting both means the browser reserves the correct box from the very first HTML parse pass; image loads later, no shift.

**Researcher to validate:** for each PNG, measure native dimensions via `sharp.metadata()` or `file` command. Generate a small JSON manifest (`scripts/img-dims.json`) so the planner can produce a single mechanical patch instead of hand-tuning each `<img>`. Same approach used for the SEO routes manifest in Phase 2.

### D-05. PNG → WebP conversion tool: `vite-imagetools` plugin
**Decision:** Install `vite-imagetools` (Vite 6 compatible, Node 24 compatible). Use its `?w=400;800&format=webp;png&as=picture` query-string syntax for each `<img>` in App.tsx that points at a PNG: change `src="/assets/landing-design/feature-models.png"` → `import featureModels from '/assets/landing-design/feature-models.png?w=400;800&format=webp;png&as=picture'` and use a small `<Picture>` wrapper component.

Apply to:
- `src/app/App.tsx` partner PNGs (4 files: canva, freepik, higgsfield, syntx)
- `src/app/App.tsx` feature card PNGs (8 files: feature-models, feature-models-en, feature-credits, feature-interface-en/ru, feature-enhance-en/ru)
- `src/app/App.tsx` steps inner PNGs (2 files: steps-inner-en, steps-inner-ru)

DO NOT convert:
- Welcome screenshots (`public/assets/welcome-*.png`) — out of scope (deferred).
- Any SVG (already optimal).
- Favicons / OG cards (Phase 1 locked them as PNG for compatibility; OG card unfurls require PNG/JPEG).
- `public/assets/landing-design/gradient-blob-shape.png` (used via CSS `background-image`, not `<img>` — separate conversion path; deferred).

**Rationale:** `vite-imagetools` integrates into Vite's existing module graph, supports per-import query strings, and outputs `<picture>` elements with srcset that the planner can drop in mechanically. Alternative `vite-plugin-image-optimizer` operates at output level (less precise control over which images get treated), and a custom `sharp` postbuild script duplicates `prerender.mjs` plumbing.

**Researcher to validate:** that `vite-imagetools` does not break Vite's existing `assetsInclude: ['**/*.svg', '**/*.csv']` config and does not interfere with the SSR entry build (`vite build --ssr scripts/entry-server.tsx`).

### D-06. Image format strategy: WebP + PNG fallback via `<picture>`, NO AVIF
**Decision:** Every converted image renders as:
```jsx
<picture>
  <source srcset="...webp" type="image/webp" />
  <img src="...png" width="N" height="N" alt="..." className="..." />
</picture>
```
WebP first, PNG fallback. No AVIF source.

**Rationale:** WebP support: 98%+ globally per caniuse, 100% on iOS 14+ and modern Android. PNG fallback covers IE11/Safari < 14 — vanishingly small slice of opten.space's traffic. AVIF adds 10-15% byte savings but iOS Safari < 16 has decoder issues and the encoder is 3-5× slower at build time. Revisit AVIF only if mobile Lighthouse < 90 after this phase.

### D-07. Build pipeline: minimal change to current chain
**Decision:** Current `package.json` build chain:
```
vite build && vite build --ssr scripts/entry-server.tsx --outDir .ssr-cache --emptyOutDir && vite build --ssr scripts/seo-routes.ts --outDir .ssr-meta && node scripts/prerender.mjs && node scripts/sitemap.mjs
```
Changes:
1. `vite build` now produces code-split chunks (transparent — D-01 just changes source imports).
2. `vite-imagetools` plugin registered in `vite.config.ts` plugin array → `vite build` produces WebP variants in `dist/assets/`.
3. `scripts/prerender.mjs` updated to (a) inject `<link rel="modulepreload">` for the main bundle (D-03) and (b) preserve any new `<picture>` markup in the prerendered React tree (it already does — `renderToString` outputs `<picture>` natively; this is a no-op).

**Rationale:** No new top-level scripts, no new build tools beyond `vite-imagetools` and (dev-only) `rollup-plugin-visualizer`. Atomic-commit pattern from Phase 1 / Phase 2 stays viable.

### D-08. Hydration mismatch protection: keep `__PRERENDER_PATH` discriminator and `createRoot` fallback unchanged
**Decision:** Do NOT touch the `__PRERENDER_PATH` marker injection in `scripts/prerender.mjs` or the path-matching detector in `src/main.tsx`. They are Phase 2 hotfix `80b16be` mitigations for React #418/#423 on SPA-fallback routes; Phase 2.1's code-split is additive — `<Suspense>` only wraps SPA-fallback `<Route>` elements that already use `createRoot`, so the hydration path-marker check still correctly routes prerendered traffic to `hydrateRoot`.

**Rationale:** Phase 2 hotfix cost a Vercel rollback when this guarantee was broken. Re-locking it as a hard constraint here.

### D-09. Measurement gate: Lighthouse mobile, on production, after deploy
**Decision:** Phase 2.1 acceptance includes running Lighthouse mobile audit against `https://opten.space/` on a deployed preview (or production), targeting Performance ≥ 90 and Cumulative Layout Shift = 0. Use `npx unlighthouse --site https://opten.space` for the run; report Performance, LCP, FID, CLS, TBT numbers in the phase VERIFICATION.md.

**Rationale:** Lighthouse mobile is the only consistently-measured metric Vercel exposes via `/inspect`. The "buttons don't respond" user complaint maps to Total Blocking Time (TBT) — the metric for "JS work that blocks main thread between FCP and TTI". Phase 2.1 should drive TBT down via smaller main chunk + parallel JS fetch.
