<objective>
Wrap every PNG-backed `<img>` in `src/app/App.tsx` with a `<Picture>` component that emits `<picture><source type="image/webp"><img src=...png></picture>` markup. The WebP source is built at `vite build` time by the `vite-imagetools@6.2.9` plugin (registered in plan 02.1-01) from the static module-level imports added in this plan.

This plan delivers Phase 2.1's perceived-load win for image payload: ~65% reduction in image bytes transferred for landing PNGs (~500 KB → ~175 KB across the 12 in-scope assets per RESEARCH.md). The visual output is pixel-identical — modern browsers pick the WebP source; legacy browsers fall back to the PNG (no behavior change for them).

**Scope (12 PNGs):**
- Partner logos ×4: `canva.png`, `freepik.png`, `higgsfield.png`, `syntx.png` (each 268×80).
- Feature card images ×7: `feature-models.png`, `feature-credits.png`, `feature-interface-ru.png`, `feature-interface-en.png`, `feature-enhance-ru.png`, `feature-enhance-en.png` (1100×424 or 1100×372).
- Steps inner ×2: `steps-inner-ru.png`, `steps-inner-en.png` (each 813×638).

**Out of scope (do NOT touch):**
- All SVG `<img>` tags (already optimal).
- `feature-models-ru.png` / `feature-models-en.png` — RESEARCH.md Open Question #2 confirms App.tsx uses `feature-models.png` (no suffix) for card1.
- `welcome-*.png` — D-05 deferred.
- `gradient-blob-shape.png` — used via CSS `background-image`, not `<img>`.

**Hard dependencies:**
- Plan 02.1-01 (imagetools plugin registered).
- Plan 02.1-02 (`Picture` component exists).
- Plan 02.1-04 (width/height attrs already on all `<img>`; `imgWidth`/`imgHeight` props already on FeatureCard).

**SC mapping:** SC-6 (no regression — visual + build green); enables image-payload reduction (perceived-load improvement, not a hard SC).
</objective>
