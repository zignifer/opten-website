# Phase 04.1 Plan 04: FaqBlock + faqPageBlock readonly widening (WR-08) — Summary

One-liner: Widened `FaqBlock.Props.items` and `scripts/seo-routes.ts#faqPageBlock` signature to accept `readonly` arrays — pre-emptive fix unblocking future TypeScript strict-mode variance checks; zero runtime change.

## What landed

**Two surgical type widenings, one atomic commit, four lines of diff total.**

### `src/app/components/FaqBlock.tsx` (1 line)
- Line 14: `items: FaqItem[];` → `items: readonly FaqItem[];`
- Body unchanged. `items.length === 0` and `items.map(...)` are both `ReadonlyArray<T>` methods.

### `scripts/seo-routes.ts` (1 line)
- Line 123: `export function faqPageBlock(items: { q: string; a: string }[], pageId: string): SchemaBlock {` → `... items: readonly { q: string; a: string }[], ...`
- Body unchanged. `items.map(...)` returns a fresh mutable array regardless of source variance.

## Why this matters

`src/content/landingFaq.ts` exports `landingFaq = { ru, en } as const`, which makes `landingFaq.ru` and `landingFaq.en` `readonly FaqItem[]`. Similarly the per-guide content (`gptImage2Guide.{lang}.faq`) flows from `as const` sources. The previous `FaqItem[]` / `{ q; a }[]` mutable signatures only accepted those readonly arrays by an implicit TypeScript variance leniency that future strict-mode tightening would close. The widening is the correct direction — callers passing `readonly` sources is the *normal* shape on this codebase, not the exception.
