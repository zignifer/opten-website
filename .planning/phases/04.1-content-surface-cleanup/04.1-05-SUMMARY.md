## How the gate works

1. Imports the SSR-bundled route manifest from
   `.ssr-meta/seo-routes.js` (same pattern as `scripts/llms.mjs` and
   `scripts/sitemap.mjs`).
2. Filters routes where `prerender !== "none"` AND
   `schema.some(b => b["@type"] === "FAQPage")`. Floor-checks the count
   at `>= 4` (post-Phase 4 expectation); throws with manifest-drift
   message if fewer.
3. For each FAQPage route:
   - Reads `dist/<route>/index.html` (with the same WR-06 path-traversal
     guard `llms.mjs` uses — meta.path rejected if it contains `..` or
     backslash; resolved outPath asserted to live under `DIST + sep`).
   - Extracts JSON-LD via `<script type="application/ld+json">([\s\S]*?)</script>`
     regex → `JSON.parse` each block → collect `mainEntity[*].name` from
     blocks where `@type === "FAQPage"`. (No HTML-entity decode needed
     for the JSON-LD path — `JSON.parse` natively handles the
     `<` / `>` / `&` escapes that `prerender.mjs::escapeJsonLd`
     writes.)
   - Extracts visible `<dt>` content from the section whose `id` matches
     `/...faq.../i` (matches both landing's `id="faq"` and GuidePage's
     `id="guide-faq"` — finding by id substring keeps the verifier robust
     to future FaqBlock callers that pick different ids per `<section
     id={id}>` prop).
   - Decodes the React-SSR entity set (`&amp;`, `&quot;`, `&lt;`, `&gt;`,
     `&nbsp;`, plus numeric decimal `&#NNN;` and numeric hex `&#xHH;` —
     mirrors `llms.mjs::extractText` for WR-07 parity), collapses
     whitespace, trims.
4. Compares the two sets per route. On mismatch throws with route +
   visible-only + schema-only diff and a hint pointing at
   `src/content/landingFaq.ts` / `scripts/seo-routes.ts` as the
   actionable source files.
5. On success logs:
   `✓ verify-faq-mainentity: 4 routes checked, 20 questions, all visible↔schema parity holds`

Temporarily mutated **only the schema input** in `scripts/seo-routes.ts`
to introduce drift (without touching visible `<dt>` source):


Captured error message:

```
  Visible-only questions (in <dt> but not in JSON-LD mainEntity):
    - Что такое Opten и как он работает?
  Schema-only questions (in JSON-LD mainEntity but not in <dt>):
    - Что такое Opten и как он работает? TEST_DRIFT
  Hint: both sides should derive from the same source array (e.g. src/content/landingFaq.ts). Check the schema input in scripts/seo-routes.ts.
```

This is the exact failure-mode the WR-09 gate was added to catch.
`scripts/seo-routes.ts` reverted after the smoke (verified with
`diff -q` against `git show HEAD:`).

### Secondary smoke (informational — confirms shared-source architecture)

Temporarily edited `src/content/landingFaq.ts` directly (the path the
plan's prompt context suggested) — appended ` TEST_DRIFT` to one RU
question's text. `npm run build` **succeeded** with the verifier
logging `4 routes checked, 20 questions, all visible↔schema parity holds`.

This is expected and correct: because `landingFaq.ts` is the single
source of truth for both `<FaqBlock items={landingFaq[lang]}/>` and
`faqPageBlock(landingFaq[lang], pageId)`, editing it updates **both
sides equally** — parity is preserved by construction. The shared-source
architecture is itself the strongest parity guarantee; the verifier
exists as defense-in-depth against future refactors that fork the
source (the scenario exercised by the primary smoke). `landingFaq.ts`
was restored to its 04.1-10 working-tree state after the smoke.

## Build-chain order (final)

```
vite build
  → vite build --ssr scripts/entry-server.tsx --outDir .ssr-cache --emptyOutDir
  → vite build --ssr scripts/seo-routes.ts --outDir .ssr-meta
  → node scripts/prerender.mjs        (emits dist/<route>/index.html × 15)
  → node scripts/sitemap.mjs          (emits dist/sitemap.xml)
  → node scripts/llms.mjs             (emits dist/llms.txt + dist/llms-full.txt)
  → node scripts/verify-faq-mainentity.mjs   ← NEW closing gate (WR-09)
```

Placement rationale (mirrors plan `<interfaces>` block): putting the
verifier **after** the artifact emitters means a parity failure still
leaves the developer with the sitemap, llms.txt, and prerendered HTML
on disk for inspection — useful when triaging.

## Deviations from plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Plan's `<section id="faq">` regex was incomplete; missed `/guides/*`**

- **Found during:** Task 1 first dry-run against `dist/`.
- **Issue:** The plan's interface block said _"guide pages have
  `id="faq"` for the FaqBlock section per `src/app/pages/GuidePage.tsx`
  convention (consistent with landing)"_ but the actual code at
  `src/app/pages/GuidePage.tsx:108` is
  `<FaqBlock items={data.faq} headingKey="guide.faqHeading" id="guide-faq" />`.
  An exact `id="faq"` regex therefore failed on both guide routes
  before the verifier ever got to the JSON-LD comparison.
- **Fix:** Generalized the section regex to
  `/<section[^>]*id="[^"]*faq[^"]*"[\s\S]*?<\/section>/i` — matches any
  id containing the substring `faq` (case-insensitive). This is also
  more future-proof: any new `FaqBlock` caller can pick an id like
  `pricing-faq`, `enterprise-faq`, etc. without breaking the gate.
- **Files modified:** `scripts/verify-faq-mainentity.mjs` only
  (no production code touched).
- **Commit:** `6595a4b` (rolled into the single atomic commit).

**2. [Rule 1 — Bug] Plan's entity decoder set missed `&#x27;`**

- **Found during:** Task 1 dry-run, second iteration (after fix #1).
- **Issue:** Plan said _"decode the minimal set (&amp; → &, &quot; → ",
  &#39; → ', &lt; → <, &gt; → >)"_ — but React 18 SSR emits the
  hexadecimal form `&#x27;` for `'`, not the decimal `&#39;`. The
  English guide FAQ "What's the max resolution GPT Image 2 supports?"
  arrived in the DOM as `"What&#x27;s the max resolution..."` and was
  flagged as drift against the JSON-LD payload (where `JSON.parse`
  decoded the apostrophe natively).
- **Fix:** Replaced the named-only decoder with a fuller set matching
  `scripts/llms.mjs::extractText` (the Phase 04.1 WR-07 baseline):
  named entities (`&amp;` `&quot;` `&lt;` `&gt;` `&nbsp;`) PLUS numeric
  decimal `&#NNN;` PLUS numeric hex `&#xHH;` with code-point validation
  and `String.fromCodePoint` fallback-to-empty on invalid inputs.
- **Files modified:** `scripts/verify-faq-mainentity.mjs` only.
- **Commit:** `6595a4b` (rolled in).
