# Phase 4.1: Content Surface Cleanup

## Goal

Close all 15 deferred findings from `04-REVIEW.md` (9 warnings + 6 info items)
in surgical, file-scoped atomic commits. No new product surface, no schema
additions, no route changes — pure remediation of issues surfaced by code
review of Phase 4.

The 2 critical findings (CR-01 JSON-LD `<script>` escape, CR-02 RU /refund
description drift) were already resolved in commit `eee40f6` during Phase 4
and are NOT part of this phase.

## Why now

The site is on production. Each finding is either customer-facing (WR-01 EN
OG cards), reputation-affecting (WR-02 $0 Free tier in Rich Results), SEO-
damaging (WR-03 breadcrumb soft-404), a real bug (WR-04 detectExtension
race, IN-02 lang-boundary anchor), or a defensive guard whose cost to add
now is much lower than the cost to add after it has masked a regression.

## Hard constraints

- **No changes to `EXTENSION_IDS`, `SUPABASE_URL`, `SUPABASE_ANON_KEY`** —
  these are mirrored in the extension and are locked by INTEGRATION-CONTRACT.
- **No changes to locked routes** (`/welcome`, `/pay`, `/success`, `/account`,
  `/dashboard/download-skill`). `/pay` keeps its synchronous Paddle CDN
  `<script>` injection and the full prerender promotion from Phase 4 D-12.
- **No regressions to Phase 4 build-time gates** — `npm run build` must keep
  producing all 14 prerendered routes with the same titles/meta/canonical/
  hreflang/JSON-LD as today; sitemap and llms.txt must keep their current
  shape (except for additive fixes called out per wave).
- **Wave 9 (PayPage.tsx) requires explicit approval** before commit — touches
  the payment surface; falls under the "extra caution around extension-coupled
  surfaces" memory rule.

## Waves

Each wave is one file scope, one atomic commit, one verification step.
Waves are ordered low-risk-first so any rollback in late waves doesn't
require unwinding earlier ones.

### Wave 1: `scripts/prerender.mjs` — OG image swap + font preload constants

**Addresses:** WR-01 (EN pages ship og-card-ru.png), IN-05 (font preload regex literals)

**Changes:**
1. Add `applyOgImage(html, meta)` helper that swaps `<meta property="og:image">`
   and `<meta name="twitter:image">` to `meta.ogImage ?? DEFAULT_OG_IMAGE`,
   escaping the URL value. Call it from the route loop.
2. Remove the obsolete comment at line 226 ("ogImage is read but not currently
   re-injected…").
3. Extract the hardcoded `/fonts/PT-Root-UI_VF.woff2` and `/fonts/Unbounded-VF.woff2`
   font preload checks into a `REQUIRED_FONT_PRELOADS` array constant at module
   top; build the regexes from it.

**Verify:**
- `npm run build` succeeds.
- `grep -l 'og-card-en.png' dist/en/**/index.html` returns all EN routes that
  have `ogImage: DEFAULT_OG_IMAGE_EN` set in seo-routes.ts (currently 7 routes).
- RU routes still grep for `og-card-ru.png`.

**Commit:** `fix(04.1): swap og:image per-route in prerender + lift font preload checks to constants (WR-01, IN-05)`

### Wave 2: `scripts/seo-routes.ts` — Product schema + breadcrumb + price safety

**Addresses:** WR-02 (Free tier in Product offers), WR-03 (breadcrumb to non-existent /guides), WR-05 (parseFloat without NaN/locale guard)

**Changes:**
1. Remove the `{name: "Free tier", price: "0", currency: "USD"}` entry from
   both `productBlock([...], ...)` calls on `/pay` and `/en/pay`. Pro is the
   Product; the free tier is acquisition copy, not an offer.
2. Collapse the 3-level breadcrumb on both guide routes to 2 levels (root →
   article), removing the `{name: "Гайды"/"Guides", url: "/guides"}` middle
   step. (Decision: do NOT create a `/guides` index page in this phase — it's
   feature work, not cleanup; a one-guide index has poor UX. Revisit when a
   second guide is added.)
3. Normalize and validate prices in `productBlock` once at function entry:
   `Number(p.price)` → throw with file/line context if not finite. Use the
   parsed number for min/max computations; keep the formatted string for
   schema output.

**Verify:**
- `npm run build` succeeds.
- `grep -c '"name": "Free tier"' dist/pay/index.html dist/en/pay/index.html`
  returns 0 for both.
- `grep -c '/guides"' dist/guides/gpt-image-2/index.html dist/en/guides/gpt-image-2/index.html`
  returns 0 for both (no breadcrumb link to bare /guides).
- A grep for `lowPrice` in dist/pay/index.html still shows the AggregateOffer
  with a real minimum price (199 RUB or whatever the single-currency case
  produces).

**Commit:** `fix(04.1): tighten Product/Offer schema + collapse guide breadcrumb to 2 levels (WR-02, WR-03, WR-05)`

### Wave 3: `scripts/llms.mjs` — traversal guard + HTML entity decoder

**Addresses:** WR-06 (path traversal), WR-07 (HTML entity decoder covers only 5 entities)

**Changes:**
1. After computing `outPath` from `meta.path`, assert
   `outPath.startsWith(DIST + sep) || outPath === resolve(DIST, "index.html")`
   and throw if it escapes. Plus reject any `meta.path` containing `..`,
   backslash, or absolute prefix at the input edge for defense in depth.
2. Extend `extractText` to decode `&#nnn;`, `&#xHHHH;`, and strip any
   remaining `&name;` named entity rather than leaking it.

**Verify:**
- `npm run build` succeeds; `dist/llms.txt` and `dist/llms-full.txt` still
  generate without errors.
- Spot-check `dist/llms-full.txt` for `&` followed by alphanumerics — should
  not appear (any future entity would be stripped, not leaked).

**Commit:** `fix(04.1): harden llms.mjs — path-traversal guard + broader HTML entity decoder (WR-06, WR-07)`

### Wave 8: `scripts/entry-server.tsx` — defense-in-depth Route assert

**Addresses:** IN-03 (no build-time check that full-tier routes are mounted in entry-server.tsx)

**Changes:**
1. In `scripts/prerender.mjs` (not entry-server.tsx itself — the assert
   belongs at the consumer, not the producer), after the SSR import is
   resolved, fail fast if any `routes.filter(r => r.prerender === "full").map(r => r.path)`
   produces a body shorter than a tiny threshold (e.g. 200 chars). That's a
   cheap proxy for "the React tree mounted something". Real-world: if
   entry-server.tsx is missing a `<Route>` for a "full" path, prerender today
   silently emits the fallback — the guard catches it.

**Verify:**
- `npm run build` succeeds on current source.
- Manually comment out the `<Route path="/about">` in entry-server.tsx
  briefly — build must fail with a clear error. Revert.

**Commit:** `fix(04.1): assert full-tier routes produce non-trivial SSR body in prerender.mjs (IN-03)`

### Wave 9: `src/app/pages/PayPage.tsx` — detectExtension race + EN anchor — REQUIRES APPROVAL

**Addresses:** WR-04 (detectExtension first-success overwritten by later "not_installed"), IN-02 (`<a href="/#faq">` crosses lang boundary for EN users)

**This wave touches the payment surface — extension-coupled. STOP before commit and ask for explicit user approval.**

**Changes:**
1. Add a `resolved` flag inside `detectExtension`; once any extension ID
   returns a positive result, subsequent callbacks no-op. Preserve all
   existing extension messaging behavior — no change to message shape, no
   change to EXTENSION_IDS, no change to retry / timeout behavior.
2. Replace hardcoded `<a href="/#faq">` with a computed
   `const homeHash = lang === "en" ? "/en/#faq" : "/#faq";` and use it on
   the anchor. Same for any other `<a href="/#…">` on PayPage.

**Verify:**
- `npm run build` succeeds.
- Manual smoke: `/pay` and `/en/pay` extension detection still works in
  isolation (extension installed + logged in → "ready"; missing → "not_installed").
  On `/en/pay`, the FAQ anchor scrolls to the EN landing FaqBlock, not RU.

**Commit:** `fix(04.1): guard detectExtension race + lang-aware FAQ anchor on /en/pay (WR-04, IN-02)`
