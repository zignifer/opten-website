### WR-01: Stale floor check in `scripts/sitemap.mjs` and `scripts/llms.mjs` not bumped for new `/en/about` route

**Files:**
- `scripts/sitemap.mjs:29-30`
- `scripts/llms.mjs:21-25`

**Issue:** Phase 4.1 Wave 12b added `/en/about` to the `seo-routes.ts`
manifest, bringing the total prerendered routes from 15 → 16. But the
floor-check sentinels in both `sitemap.mjs` and `llms.mjs` still read
`< 15`. This means if someone accidentally deletes the new `/en/about`
entry (or one of the old EN siblings) the build will silently emit a
15-route sitemap / llms.txt without failing. The whole point of the
floor check is to catch this. The post-04.1 expected count is 16, not 15.

`sitemap.mjs:30` error message: `"expected at least 15 routes (6 RU + 6 EN + /about + 2 guide siblings per Phase 4 D-01/D-06)"` — the math itself is also wrong post-04.1 (6 RU + 7 EN, not 6 RU + 6 EN + /about). And `llms.mjs:23` error message: `"expected at least 15 prerendered routes (12 baseline + /about (04-05) + 2 guide siblings (04-06))"` — also drifted by one.

**Fix:**
```js
// scripts/sitemap.mjs
if (sitemapRoutes.length < 16) {
  throw new Error(`sitemap.mjs: expected at least 16 routes (7 RU + 7 EN + 2 guide siblings per Phase 4 D-01/D-06 + Phase 4.1 B-03 /en/about), got ${sitemapRoutes.length}. Manifest mis-loaded or entries missing?`);
}

// scripts/llms.mjs
if (prerenderedRoutes.length < 16) {
  throw new Error(`llms.mjs: expected at least 16 prerendered routes (12 baseline + /about (04-05) + 2 guide siblings (04-06) + Phase 4.1 /en/about), got ${prerenderedRoutes.length}. Manifest mis-loaded or routes missing?`);
}
```

(Note: `scripts/verify-faq-mainentity.mjs:29` floor of 4 FAQPage routes is correct — Phase 4.1 did not add a new FAQPage route.)

### IN-01: `productBlock` price normalization does not reject locale-comma input ("199,99") or trailing-space whitespace

**File:** `scripts/seo-routes.ts:160-168`

**Issue:** The `Number(p.price)` call correctly rejects non-numeric
input (`"199 руб"` → `NaN` → throws), but `Number("199,99")` also
returns `NaN` so locale-comma input fails as expected. However:
- `Number(" 199 ")` returns `199` (silent whitespace trim). Future translator typo with trailing space passes silently.
- `Number("199.00")` and `Number("199")` both yield `199` — lowPrice/highPrice resolution then uses `find(p => p._n === 199)` which picks the FIRST input matching that numeric value. With two equal-numeric tiers of different formats, the picked formatted string is arbitrary order-dependent. Today there's only one tier per currency so this doesn't bite, but it's a latent fragility if a future Pro Quarterly tier shares a numeric with Monthly.

**Fix (defensive):**
```ts
const parsed = plans.map((p) => {
  if (typeof p.price !== "string" || p.price !== p.price.trim()) {
    throw new Error(`productBlock(${pageId}): price "${p.price}" for plan "${p.name}" has leading/trailing whitespace or is non-string`);
  }
  const n = Number(p.price);
  if (!Number.isFinite(n)) {
    throw new Error(`productBlock(${pageId}): non-finite price "${p.price}" for plan "${p.name}" — expected a number-parseable string (e.g. "199" or "2.99")`);
  }
  return { ...p, _n: n };
});
```

### IN-02: `applyHeroPreloadGuard` regex fragile to attribute-order changes

**File:** `scripts/prerender.mjs:251-259`

**Issue:** The regex `<link\s+rel="preload"\s+href="${...}"[^>]*as="font"`
hardcodes the attribute order `rel="preload" href=... as="font"`. The
HTML template (`index.html:67-68`) ships them in exactly this order
today, so the check passes. But the guard's claim is "Phase 2.2
regression — fail loudly if anyone deletes those preload tags." A
future formatter (or a Figma re-export) could reorder attributes to
`<link href=... rel="preload" as="font" />`, silently breaking the
guard without removing the preload — the LCP regression would slip
through and the guard would produce a confusing false-negative error.

**Fix:** Use a position-independent check:
```js
function applyHeroPreloadGuard(html, meta) {
  for (const fontPath of REQUIRED_FONT_PRELOADS) {
    const linkRe = new RegExp(`<link\\b[^>]*\\bhref="${escapeRegex(fontPath)}"[^>]*>`, "g");
    const links = [...html.matchAll(linkRe)];
    const hasPreloadFont = links.some(([m]) => /\brel="preload"/.test(m) && /\bas="font"/.test(m));
    if (!hasPreloadFont) {
      throw new Error(`prerender(${meta.path}): missing font preload ${fontPath} — Phase 2.2 regression. See .planning/phases/04-content-surface/04-LCP-AUDIT.md.`);
    }
  }
  return html;
}
```

**File:** `src/app/pages/PayPage.tsx:195-231`

**Issue:** If the Chrome runtime is available (`chrome?.runtime?.sendMessage` truthy) but neither sendMessage callback EVER fires (e.g. extension hung, extension installed but background script not yet alive on cold start, network of some kind), the `detectExtension` flow leaves `extStatus = "detecting"` forever. ### IN-07: `extractText` in `llms.mjs` strips named entities AFTER decoding 5 known names — order is correct, but the strip is destructive

**File:** `scripts/llms.mjs:100-102`

**Issue:** The named-entity strip (`replace(/&[a-zA-Z][a-zA-Z0-9]*;/g, "")`)
runs after the 5 known named entities (`&nbsp;`, `&amp;`, `&lt;`,
`&gt;`, `&quot;`, `&#39;`) are decoded. Any remaining named entity
(e.g. `&copy;`, `&hellip;`, `&mdash;`) is replaced with empty string —
the character is LOST from `llms-full.txt`. The author's body copy
(`src/content/about.tsx` line 99) uses literal "₽" (U+20BD) and the
Russian copy uses Cyrillic directly, not entities, so the entity
exposure on the bundled HTML is mostly limited to React's SSR output of
apostrophes (`&#x27;`) and quotes — which the numeric decoders DO
handle. But the `&copy;` glyph in the footer copyright line
(`"footer.copyright": "© 2026 Opten · ИП Воронежцев В.П."`) is the
literal `©` UTF-8 character in the JSON, not the entity, so it survives.

The risk: if a future contributor inserts `&hellip;` or `&copy;` directly into a JSX child, llms-full.txt would silently drop the glyph rather than emit a fallback. Comment says "Drop any remaining named entity rather than leak it as raw text." — defensible choice, but worth documenting as a deliberate lossy decode.

**Fix (optional):** Either keep the silent strip with a clearer comment, OR replace unknown named entities with a placeholder (e.g. `?`) so the loss is visible in the output. Current code is correct as documented.
