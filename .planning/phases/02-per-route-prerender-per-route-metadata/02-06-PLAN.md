
Chain order (deliberate):
1. `vite build` — produces `dist/index.html`, hashed asset bundles, copies `public/*` to `dist/`.
2. `vite build --ssr scripts/entry-server.tsx --outDir .ssr-cache --emptyOutDir` — produces `.ssr-cache/entry-server.mjs`. `--emptyOutDir` is on this call so it cleans `.ssr-cache/` once at the start of the SSR phase.
3. `vite build --ssr scripts/seo-routes.ts --outDir .ssr-cache` — produces `.ssr-cache/seo-routes.mjs`. `--emptyOutDir` is OMITTED so this call does not wipe `.ssr-cache/entry-server.mjs` from step 2.
4. `node scripts/prerender.mjs` — reads `dist/index.html` + `.ssr-cache/*.mjs`, emits 6 per-route HTML files into `dist/`.
5. `node scripts/sitemap.mjs` — reads `.ssr-cache/seo-routes.mjs`, overwrites `dist/sitemap.xml` with the lastmod-enabled version.

Why two separate SSR builds: Vite SSR build accepts a single entry per invocation. Compiling both `entry-server.tsx` and `seo-routes.ts` requires two calls. (Alternatively a `lib` config could compile both in one pass, but that would require a `vite.ssr.config.ts` file — adds a dependency surface area we deliberately avoid per D-08.)
