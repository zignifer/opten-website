# Phase 2 Plan 05: Postbuild Sitemap Emitter Summary

## One-liner

Node ESM postbuild script that reads the `.ssr-meta/seo-routes.js` manifest, filters the 6 non-"none" routes, and emits `dist/sitemap.xml` with a uniform build-time `<lastmod>` on every entry (GEO-B-3 / D-04).

## What Was Done

Created `scripts/sitemap.mjs` — the XML postbuild emitter that closes GEO-B-3.

### Script structure

- ESM `.mjs`, Node 20+ top-level await
- `__dirname` recovered via `dirname(fileURLToPath(import.meta.url))`
- `pathToFileURL()` wrapping for Windows-compatible dynamic `import()`
- Constants: `ROOT`, `DIST`, `MANIFEST_BUNDLE` (`.ssr-meta/seo-routes.js`), `BUILD_DATE`
- Dynamic import of manifest at script start
- Filter: `routes.filter(r => r.prerender !== "none")` → 6 routes
- Defensive guard: throws loudly if filter yields 0 routes (manifest regression protection)
- Template-literal XML construction (no XML library needed — 6 routes, fixed schema)
- `priority.toFixed(1)` produces `1.0`, `0.8`, `0.6`, `0.3` matching Phase 1 format

### Routes emitted

| Route | Priority | changefreq |
|-------|----------|------------|
| `https://opten.space/` | 1.0 | weekly |
| `https://opten.space/pay` | 0.8 | monthly |

| `https://opten.space/privacy` | 0.3 | yearly |
| `https://opten.space/terms` | 0.3 | yearly |
| `https://opten.space/refund` | 0.3 | yearly |

### D-04 compliance

All 6 `<lastmod>` values are the same ISO date (`2026-05-14` in the test run), computed once at script start via `new Date().toISOString().split("T")[0]`. Every push to main is a batch release, so uniform date is the correct granularity.

### Separation of concerns

| Script | Responsibility |
|--------|---------------|
| `scripts/prerender.mjs` (Plan 02-04) | HTML emission — per-route `dist/{route}/index.html` |
| `scripts/sitemap.mjs` (this plan) | XML emission — `dist/sitemap.xml` with `<lastmod>` |

Both scripts read from `.ssr-meta/seo-routes.js`. Plan 02-06 wires both into `npm run build`.

### Manual invocation chain (for Plan 02-06 wiring reference)

```sh
npx vite build
npx vite build --ssr scripts/entry-server.tsx --outDir .ssr-cache --emptyOutDir
npx vite build --ssr scripts/seo-routes.ts    --outDir .ssr-meta
node scripts/prerender.mjs
node scripts/sitemap.mjs
```

### Phase 1 invariants preserved

- `public/sitemap.xml` not modified — it is the static fallback for graceful degradation
- `dist/sitemap.xml` is generated by this script and overrides the Vite-copied fallback
- Route set is an exact match of Phase 1's 6-route sitemap (no route added or removed)

## Deviations from Plan

### Inherited from Plan 02-04

**1. [Rule 3 - Blocking, inherited] .ssr-cache → .ssr-meta for manifest bundle**
- **Found during:** Plan 02-04 (Vite 6 empties outDir on every SSR build)
- **Issue:** Plan 02-05 spec said `MANIFEST_BUNDLE = resolve(ROOT, ".ssr-cache", "seo-routes.mjs")`. This is stale — 02-04 discovered that two SSR builds targeting the same outDir causes Vite 6 to erase the first bundle when running the second. Fix was to use separate outDirs: `.ssr-cache/` for `entry-server.tsx`, `.ssr-meta/` for `seo-routes.ts`.
- **Fix applied:** `MANIFEST_BUNDLE = resolve(ROOT, ".ssr-meta", "seo-routes.js")` (note also `.js` extension, not `.mjs` — Vite 6 SSR bundle output extension)
- **Files modified:** `scripts/sitemap.mjs`
- **Commit:** fad74f8

**2. [Rule 3 - Blocking, inherited] pathToFileURL() for Windows dynamic import**
- **Found during:** Plan 02-04 (Node ESM loader rejects bare `C:\` absolute paths)
- **Fix applied:** `await import(pathToFileURL(MANIFEST_BUNDLE).href)`
- **Files modified:** `scripts/sitemap.mjs`
- **Commit:** fad74f8

### Impact on Plan 02-06

Plan 02-06 (wiring the build chain into `package.json`) must use:


as the last step, after `node scripts/prerender.mjs`. The full chain is:

